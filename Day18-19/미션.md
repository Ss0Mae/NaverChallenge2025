# **학습 목표**

- TCP/UDP 방식으로 동작하는 클라이언트와 서버를 직접 설계하고 구현할 수 있다.
- 필요한 요청과 응답 메시지 구조를 JSON으로 주고 받을 수 있다.
- TCP 연결을 관리하고, 유니캐스트 동작을 구현할 수 있다.
- UDP 주소를 활용해서 브로드캐스트 동작을 구현할 수 있다.

# **사전지식**

HTTP Request와 Response 구조, JSON 데이터 방식에 대해 알고 충분히 활용한다.

# **기능요구사항**

### **A. 고객 요청 요구사항**

- 서버가 받은 모든 요청에는 적절한 응답을 보내준다. (아무런 반응이 없으면 안된다.)
- 서버는 여러 클라이언트가 동시에 연결할 수 있어야 한다.
- login 요청을 받은 후에도 연결을 유지해야 한다.

### **1. login**

- 고객 클라이언트는 서버에 처음 소켓 연결하고 나면 **`login`** 요청을 보내야 한다.
    - login 데이터로 **`campId`**를 문자열로 전달한다.
    - 캠퍼아이디 범위는 **`S001`** 에서 **`S256`** 까지로 제한한다.
    - 범위를 벗어나는 경우는 응답으로 실패를 알려줘야 한다.
    - 클라이언트는 캠퍼아이디가 범위를 벗어난 경우 재입력할 수 있어야 한다.
    - 이미 login 한 상태에서는 login을 할 수 없다.
- **`login`**을 할 때 서버에서는 쇼핑 그룹을 할당한다.
    - 그룹은 최대 4명까지 가능해서 5명부터는 다른 그룹에 할당한다.
    - 그룹에서 빠져나가서 4명보다 작은 경우에는 재할당할 수 있다. (어떤 방식으로 구현해도 된다.)
- **`login`** 응답으로는 배정된 쇼핑그룹 번호를 정수형으로 알려준다.

### **2. logout**

- 특정 클라이언트가 **`logout`** 요청을 보내면 이전에 **`login`** 했던 쇼핑 그룹에서 퇴장한다.
- 만약 해당 그룹에 다른 캠퍼가 한 명이라도 남아있다면, 누군가 퇴장했다는 것을 알려준다.
- 클라이언트는 logout 요청 이후 응답을 받으면 연결을 끊는다.
- 서버는 클라이언트가 logout을 하지 않고 TCP 연결이 끊어져도 해당 연결은 logout으로 동일하게 처리한다.

### **3. catalog**

- 체크인 이후에는 **`catalog`**요청을 보내서 등록된 상품 목록을 받을 수 있다.
- 서버에서는 값을 구분해서 상품 목록을 응답한다.
    - 상품 목록은 상품id, 상품명, 재고 개수를 포함한다.
- 상품 목록은 최소한 15개 이상 보낸다. 만약 등록된 상품이 15개 미만인 경우는 에러값을 응답한다.

### **4. snapchat**

- 채팅 요청을 보내면 같은 쇼핑 그룹에 있는 사람들과 스냅채팅을 할 수 있다.
- 요청 데이터로는 **`maxCount`**를 정수형으로 보낼 수 있다.
- 사람들과 주고받을 수 있는 메시지 개수는 **`maxCount`**보다 클 수 없다.
- 최대 메시지 개수를 넘어가면 누군가 메시지를 보내도 서로에게 전달되지 않는다.

### **5. message**

- snapchat 도중에 메시지 요청을 보내면 서버는 고객 클라이언트에게 브로드캐스트한다. 로그인하지 않거나 로그아웃 상태 클라이언트도 모두 받아야 한다.
- 메시지 요청의 데이터는 **`text`**로 문자열을 받을 수 있다.

### **6. finish**

- snapchat 요청을 보낸 동일한 사용자가 finish를 보내면 채팅을 멈출 수 있다.
- 그 이후에는 다른 사용자가 다시 snapchat 시작할 수 있고 채팅 중에는 다시 시작할 수 없다

### **7. buy**

- 카타로그로 받았던 상품을 구매할 수 있다.
- 요청 데이터로 구매하려는 **`상품id`** 와 **`개수`** 를 함께 보내야 한다.
- 만약 재고 개수가 구매 개수보다 작으면 서버에서 에러를 리턴할 수 있다.
- 구매 후에는 재고 개수가 줄어들어야 한다.

### **B. MD 요청 요구사항**

- 서버가 받은 모든 요청에는 적절한 응답을 보내준다. (아무런 반응이 없으면 안된다.)
- 서버는 MD 연결을 하나만 유지해야 한다.

### **1. auth**

- MD 클라이언트는 서버에 처음 소켓 연결하고 나면 **`auth`** 요청을 보내야 한다.
- 보안상 주어진 토큰을 함께 보내야 한다. 토큰은 서버와 클라이언트만 알고 있는 해시 함수를 활용한다.

### **2. add**

- MD 클라이언트에서는 상품 등록을 위해 **`add`** 요청을 보내야 한다.
- 매개변수로 **`상품 id`** , **`상품명`**, **`재고 개수`**를 전달한다.
- 만약 기존에 있는 상품이면 재고만 늘어나고 없으면 새로 등록되어야 한다.

### **3. broadcast**

- 메시지 요청을 보내면 서버는 UDP 방식으로 실행중인 모든 고객 클라이언트에게 브로드캐스트한다. 로그인하지 않거나 로그아웃 상태 클라이언트도 모두 받아야 한다.
- 메시지 요청의 데이터는 **`text`**로 문자열을 받을 수 있다.

### **4. logout**

- MD 클라이언트가 **`logout`** 요청을 보내면 MD 연결을 종료한다.
- 서버는 MD 클라이언트가 logout을 하지 않고 TCP 연결이 끊어져도 해당 연결은 logout으로 동일하게 처리한다.

# **프로그래밍 요구사항**

- JS에서 제공하는 네트워크 라이브러리를 활용한다.
    - 노드 net.Socket [공식문서](https://nodejs.org/api/net.html)
- TCP 방식으로 연결하는 고객용과 MD용 클라이언트와 서버를 모두 구현해야 한다.
- 서버는 2025포트를 사용한다.
- 전송하는 요청과 응답은 HTTP 규격을 비슷하게 만들기를 추천하고, 데이터 구조는 JSON 형식을 활용한다.
    - 꼭 HTTP와 동일하지는 않아도 되고, 사람이 읽어서 요청과 응답을 구분할 수 있으면 된다.

### **클라이언트 요구사항**

- 고객/MD 클라이언트는 요청을 입력할 수 있는 비동기 입력 화면이 있어야 한다.
- 고객/MD 클라이언트는 서버로 보낸 요청과 서버에서 받은 응답, 그리고 서버에서 보내는 메시지를 출력한다.
- 고객/MD 클라이언트는 **`login`** 성공한 시각을 변수에 기록한다.
