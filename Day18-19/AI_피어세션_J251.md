# AI 페어 프로그래밍: 실시간 코드 리뷰 및 디버깅 파트너로 활용하기

**학습 키워드:** #페어프로그래밍 #실시간디버깅 #통합테스트 #리팩토링

## 1. 시도 방법 및 배경

이번 Day18-19 미션에서는 TCP/UDP 서버와 클라이언트를 구현하며 복잡한 비동기 로직과 여러 모듈 간의 상호작용을 다루어야 했습니다. 특히 통합 테스트 과정에서 예측하지 못한 오류들이 발생했을 때, 기존의 문제 해결 방식(오류 메시지 검색, 코드 재검토)만으로는 시간이 많이 소요될 것이라 판단했습니다.

이에, AI를 단순한 정보 검색 도구가 아닌 **실시간으로 소통하는 페어 프로그래밍 파트너**로 활용하는 새로운 접근법을 시도했습니다. 구체적인 방법은 다음과 같습니다.

1.  **문제 상황 실시간 공유:** 테스트 실행 후 발생한 오류 메시지, 스택 트레이스, 그리고 관련 코드를 그대로 AI에게 전달합니다.
2.  **AI의 1차 진단 및 해결책 제안 받기:** AI가 제공하는 분석과 해결책을 검토합니다.
3.  **해결책 적용 및 결과 공유:** 제안된 해결책을 코드에 적용한 후, 변경된 코드와 새로운 테스트 결과를 다시 AI에게 공유하며 추가적인 문제를 함께 논의합니다.
4.  **지속적인 리팩토링:** 문제가 해결된 후에도, 테스트 결과에 나타난 비효율(예: 과도한 로그 출력)을 공유하며 코드 품질 개선에 대한 피드백을 요청합니다.

이러한 상호작용을 통해, 혼자서는 놓칠 수 있는 부분을 AI 파트너가 짚어주고 함께 해결해나가는 과정을 경험하고자 했습니다.

## 2. 시도 내용 및 결과

### 문제 1: 통합 테스트 `TypeError` 발생

-   **나의 시도:** `integration.test.js`를 작성하여 MD 클라이언트가 상품을 등록하고 고객 클라이언트가 해당 상품 목록을 조회하는 시나리오를 테스트했지만, `TypeError: Cannot read properties of undefined (reading 'address')` 오류가 발생했습니다. 오류 로그 전체를 AI에게 공유하며 원인 분석을 요청했습니다.

-   **AI의 진단 및 해결책:**
    -   AI는 스택 트레이스를 분석하여 `testServer.address()` 호출 시 `testServer` 객체가 `undefined`임을 정확히 지적했습니다.
    -   원인으로 `Server.js`의 `listen` 메서드가 서버 인스턴스를 반환하지 않는다는 점을 정확히 짚어냈습니다.
    -   `listen` 메서드에 `return this.server.listen(port, callback);`과 같이 `return` 문을 추가하는 구체적인 해결책을 제시했습니다.

-   **결과:** AI의 제안대로 코드를 수정하자 `TypeError`가 성공적으로 해결되었습니다.

### 문제 2: "Jest did not exit" 비동기 처리 오류

-   **나의 시도:** 첫 번째 오류를 해결하자, 이번에는 테스트는 통과했지만 `Jest did not exit one second after the test run has completed.` 라는 경고가 발생했습니다. 이는 비동기 작업이 제대로 종료되지 않았음을 의미합니다. 이 경고 메시지와 관련 코드(`BaseClient.js`, `integration.test.js`)를 AI에게 공유하며 해결을 요청했습니다.

-   **AI의 진단 및 해결책:**
    -   AI는 이 문제가 테스트 종료 후에도 소켓 연결 같은 비동기 작업이 남아있기 때문에 발생한다고 설명했습니다.
    -   `disconnect()` 함수가 소켓 종료를 요청하기만 할 뿐, 실제로 연결이 끊어질 때(`close` 이벤트)까지 기다리지 않는다는 핵심 원인을 정확히 분석했습니다.
    -   해결책으로 `BaseClient.js`의 `disconnect` 함수가 `Promise`를 반환하고 `close` 이벤트 발생 시 `resolve` 하도록 수정하고, `integration.test.js`에서는 `await` 키워드로 `disconnect()` 호출을 기다리는 두 단계의 명확한 해결책을 제시했습니다.

-   **결과:** AI가 제안한 대로 코드를 수정하자, Jest가 깔끔하게 종료되며 모든 테스트가 경고 없이 통과했습니다.

### 문제 3: 테스트 결과의 과도한 로그 출력

-   **나의 시도:** 모든 테스트가 성공적으로 통과했지만, 테스트 결과에 애플리케이션의 모든 `console.log`가 출력되어 테스트의 성공 여부를 한눈에 파악하기 어려웠습니다. 이 지저분한 출력 결과를 AI에게 보여주며 개선 방향에 대한 의견을 물었습니다.

-   **AI의 진단 및 해결책:**
    -   AI는 이 문제가 테스트의 본질적인 결과 확인을 방해한다고 공감했습니다.
    -   `process.env.NODE_ENV` 환경 변수를 확인하여, `'test'` 환경일 때는 로그를 출력하지 않도록 각 모듈의 `console.log` 부분을 조건문으로 감싸는 리팩토링 방안을 제안했습니다.

-   **결과:** 현재 이 제안을 바탕으로 코드를 개선하는 작업을 진행 중입니다. 이 방식을 적용하면 테스트 결과가 훨씬 깔끔해져서 CI/CD 환경에서도 유용할 것으로 기대됩니다.

## 3. 느낀 점 및 앞으로의 활용 계획

AI와 페어 프로그래밍을 하는 경험은 기대 이상으로 효율적이었습니다. 특히 **명확한 오류 메시지와 스택 트레이스가 있는 경우, AI는 매우 빠르고 정확하게 원인을 분석하고 해결책을 제시**했습니다. 단순히 "오류가 났어"라고 묻는 대신, "이런 코드가 있고, 이런 오류가 발생했어. 어떻게 생각해?"라고 구체적인 컨텍스트를 제공하는 것이 핵심이었습니다.

이번 경험을 통해 얻은 가장 큰 인사이트는 **AI를 '검색 엔진'이 아닌 '개발 파트너'로 대할 때 그 잠재력이 극대화된다**는 것입니다. 앞으로의 미션에서도 다음과 같이 AI를 적극적으로 활용할 계획입니다.

-   **디버깅 파트너:** 복잡한 버그 발생 시, 관련 코드와 로그를 공유하며 함께 원인을 추적하는 '실시간 디버깅 세션'을 진행할 것입니다.
-   **코드 리뷰어:** 기능 구현 후, 특정 모듈의 코드를 보여주며 "더 좋은 구조는 없을까?", "이 코드의 잠재적인 문제점은 무엇일까?"와 같은 질문을 통해 코드 품질을 개선하는 데 활용할 것입니다.
-   **리팩토링 설계자:** 재사용성을 높이거나 유지보수성을 개선하고 싶을 때, 관련 코드들을 제시하고 구조 변경에 대한 아이디어를 얻는 데 사용할 것입니다.

이러한 접근 방식은 단순히 문제 해결 시간을 단축하는 것을 넘어, 제가 미처 생각하지 못했던 새로운 관점이나 개선 아이디어를 얻는 데 큰 도움이 될 것이라 확신합니다.
