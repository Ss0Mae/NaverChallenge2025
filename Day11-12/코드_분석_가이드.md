# 🧑‍💻 `my-git.js` 코드 분석 가이드

안녕하세요! 이 문서는 저희가 Day13-14 미션으로 구현한 `my-git.js`의 전체 코드 흐름과 각 함수의 핵심 로직을 피어세션에서 쉽게 이해하고 토론하기 위해 작성되었습니다.

## 📂 전체 구조

우리 프로그램은 사용자가 입력한 커맨드라인 인자를 받아 특정 기능을 수행하는 간단한 CLI(Command-Line Interface) 구조를 가집니다.

```javascript
const command = process.argv[2]; // 예: 'init', 'add'
const argument = process.argv[3];  // 예: 'file.txt', '첫 커밋'

switch (command) {
  case 'init': init(); break;
  case 'add': add(argument); break;
  // ... 다른 명령어들
}
```

- `process.argv`를 통해 사용자가 입력한 명령어를 받아옵니다.
- `switch` 문을 사용해 명령어에 따라 적절한 메인 함수(`init`, `add` 등)를 호출합니다.

이제 각 명령어의 핵심 로직을 자세히 살펴보겠습니다.

---

## 1️⃣ `init`: 저장소의 탄생

`init`은 모든 작업의 시작점으로, Git이 버전 관리를 할 수 있는 기본 환경을 구축합니다.

- **핵심 역할**: `.git` 디렉토리와 그 안의 필수 하위 디렉토리, 그리고 `HEAD` 파일을 생성합니다.
- **주요 로직**:
    1.  `fs.existsSync`로 `.git` 디렉토리가 이미 있는지 확인하여, 있다면 중복 생성을 막고 메시지를 출력하며 종료합니다.
    2.  `fs.mkdirSync`를 사용해 `.git` 폴더와 그 안에 `objects`, `refs/heads` 폴더를 생성합니다. 이때 `mode: 0o755` 옵션으로 실제 Git과 유사하게 디렉토리 권한을 설정합니다.
    3.  `fs.writeFileSync`로 `.git/HEAD` 파일을 생성하고, `ref: refs/heads/main`이라는 내용을 써넣습니다. 이는 저장소의 현재 상태가 `main` 브랜치를 가리키고 있음을 의미합니다.

## 2️⃣ `add`: 변경점을 스테이징 영역으로

`add`는 우리 프로그램에서 가장 복잡하고 중요한 부분 중 하나입니다. 파일의 현재 상태를 스냅샷(blob 객체)으로 만들고, 이 정보를 다음 커밋에 포함시키기 위해 스테이징 영역(`.git/index`)에 기록합니다.

이 작업은 크게 **객체화**와 **인덱싱** 두 단계로 나뉩니다.

### 가. `hashObject`: 파일 내용을 Blob 객체로

이 함수는 파일을 받아 Git 데이터베이스에 저장 가능한 `blob` 객체로 변환하고, 그 객체의 고유 ID(해시)를 반환합니다.

- **핵심 역할**: `파일 내용` → `Blob 객체` → `SHA-1 해시` → `.git/objects`에 저장
- **주요 로직**:
    1.  `fs.readFileSync`로 파일 내용을 읽습니다.
    2.  Git의 규칙에 따라 `blob [콘텐츠 길이]\0[콘텐츠]` 형식의 헤더를 만듭니다. `\0`은 헤더와 내용을 구분하는 null 바이트입니다.
    3.  헤더와 파일 내용을 `Buffer.concat`으로 합쳐 완전한 `blob` 객체를 만듭니다.
    4.  `crypto.createHash('sha1')`로 객체 전체의 SHA-1 해시를 계산합니다.
    5.  `zlib.deflateSync`로 객체를 압축합니다. Git은 공간 효율을 위해 모든 객체를 압축 저장합니다.
    6.  해시의 앞 2자리를 디렉토리 이름, 나머지 38자리를 파일 이름으로 사용하여 압축된 객체를 `.git/objects`에 저장합니다. 예를 들어 해시가 `e2e67d...`라면, `.git/objects/e2/e67d...` 경로에 저장됩니다.

### 나. `updateIndex`: 인덱스(스테이징 영역) 업데이트

`hashObject`로 생성된 `blob` 객체의 정보를 `.git/index` 파일에 기록하는 과정입니다. 이 `index` 파일이 바로 스테이징 영역의 실체입니다.

- **핵심 역할**: `add`된 파일의 메타데이터와 `blob` 해시를 `.git/index` 파일에 기록/수정합니다.
- **주요 로직**:
    1.  **`readIndex()`**: 먼저 기존의 `.git/index` 파일을 읽어 현재 스테이징된 파일 목록을 가져옵니다. (자세한 내용은 아래 `index` 파일 심층 분석에서 설명)
    2.  `fs.statSync`로 `add`하려는 파일의 현재 상태(생성/수정 시간, 권한, 소유자 정보 등)를 가져옵니다.
    3.  이 메타데이터와 `hashObject`로부터 받은 해시를 조합하여 새로운 인덱스 엔트리 객체를 만듭니다.
    4.  `entries.findIndex`로 파일이 이미 인덱스에 있는지 확인하고, 있다면 새 엔트리로 교체하고 없다면 목록에 추가합니다.
    5.  **`writeIndex()`**: 변경된 엔트리 목록 전체를 다시 바이너리 형식으로 만들어 `.git/index` 파일에 덮어씁니다.

### 💡 `index` 파일 심층 분석 (`readIndex` & `writeIndex`)

`.git/index`는 텍스트가 아닌, 정해진 규칙에 따른 바이너리 파일이라 다루기가 매우 까다롭습니다.

- **`readIndex`의 동작 원리**:
    1.  파일을 `Buffer`로 읽어옵니다.
    2.  **헤더(12바이트) 파싱**:
        -   `DIRC` 시그니처, 버전 정보, 엔트리 개수를 읽어 유효한 파일인지 확인합니다.
    3.  **엔트리 반복 파싱**:
        -   헤더에서 읽은 엔트리 개수만큼 반복합니다.
        -   각 엔트리는 **고정된 62바이트**와 **가변 길이의 파일 경로**로 이루어져 있습니다.
        -   `buffer.readInt32BE(offset)` 같은 메서드를 사용해 정해진 위치에서 `ctime`(생성시간), `mtime`(수정시간), `mode`(권한), `size`, 그리고 가장 중요한 **`sha1`(blob 해시)** 등의 정보를 정확히 잘라내어 파싱합니다.
        -   파일 경로는 길이가 가변적이므로, `flags`에서 길이를 알아내어 그만큼 잘라냅니다.
        -   Git은 성능을 위해 각 엔트리를 8바이트 단위로 정렬(padding)하므로, 다음 엔트리로 넘어가기 전에 패딩 길이를 계산하여 건너뜁니다.

- **`writeIndex`의 동작 원리**: `readIndex`의 과정을 정확히 거꾸로 수행합니다.
    1.  헤더 정보를 담은 12바이트 `Buffer`를 먼저 만듭니다.
    2.  각 엔트리 객체를 순회하며, `entryBuffer.writeInt32BE(...)` 등을 사용해 모든 메타데이터를 바이너리 형식으로 버퍼에 기록합니다.
    3.  모든 엔트리 버퍼를 하나로 합칩니다.
    4.  헤더와 합쳐진 엔트리 버퍼 전체의 SHA-1 해시를 계산하여 **체크섬(checksum)**을 만듭니다.
    5.  `[헤더 버퍼] + [모든 엔트리 버퍼] + [체크섬 해시]`를 최종적으로 합쳐 `.git/index` 파일에 씁니다.

## 3️⃣ `commit`: 역사의 한 페이지 기록

`commit`은 `index`에 기록된 현재 스테이징 상태를 바탕으로, 영구적인 스냅샷을 만들어 저장소에 기록합니다.

- **핵심 역할**: `index` → `tree` 객체 → `commit` 객체 → 브랜치 포인터 이동
- **주요 로직**:
    1.  **`writeTree()`**: `index` 파일의 내용을 바탕으로 `tree` 객체를 생성합니다. `tree` 객체는 특정 시점의 디렉토리 구조를 나타냅니다. 각 엔트리는 `<mode> <path>\0<sha1>` 형식이며, 이 목록 전체를 `tree` 객체로 만들어 `.git/objects`에 저장합니다.
    2.  **`getHeadCommit()`**: 현재 브랜치의 마지막 커밋(즉, 새로 만들 커밋의 부모) 해시를 알아내기 위해 `.git/HEAD`와 `.git/refs/heads/...` 파일을 읽습니다.
    3.  **커밋 객체 내용 생성**: `tree` 해시, `parent` 해시, 작성자 정보, 현재 시간, 그리고 커밋 메시지를 정해진 형식의 텍스트로 조합합니다.
    4.  이 텍스트를 `commit` 객체로 만들어(`commit [크기]\0[내용]`) 해시를 계산하고, 압축하여 `.git/objects`에 저장합니다.
    5.  **`updateHead()`**: 현재 브랜치 파일(예: `.git/refs/heads/main`)의 내용을 방금 만든 새 커밋의 해시로 덮어써서, 브랜치가 최신 커밋을 가리키도록 합니다.
    6.  마지막으로, 커밋이 완료되었으므로 `fs.unlinkSync`로 `.git/index` 파일을 삭제하여 스테이징 영역을 깨끗하게 비웁니다.

## 4️⃣ `log` & `branch`: 역사 조회와 가지치기

### `log`: 커밋 여정 돌아보기

- **핵심 역할**: `HEAD`부터 시작해 부모 커밋을 계속 따라가며 커밋 히스토리를 출력합니다.
- **주요 로직**:
    1.  `getHeadCommit()`으로 최신 커밋 해시를 가져옵니다.
    2.  `while` 루프를 돌며 현재 커밋 해시가 유효한 동안 반복합니다.
    3.  **`readObject()`**: 해시를 이용해 `.git/objects`에서 커밋 객체 파일을 읽어 압축을 풉니다. 이 함수는 어떤 타입의 객체든 읽을 수 있는 범용 함수입니다.
    4.  커밋 객체의 내용을 파싱하여 작성자, 날짜, 메시지를 추출하고 보기 좋게 출력합니다.
    5.  내용에서 `parent` 라인을 찾아 다음으로 탐색할 부모 커밋의 해시를 얻습니다. 부모가 없으면 루프가 종료됩니다.

### `branch`: 새로운 작업 흐름 만들기

- **핵심 역할**: 브랜치를 보여주거나, 특정 커밋을 가리키는 새로운 이름표(브랜치)를 만듭니다.
- **주요 로직**:
    -   **목록 보기 (`branch` 인자 없음)**: `.git/refs/heads` 디렉토리의 모든 파일명을 읽어옵니다. `getHeadRef`로 현재 브랜치를 알아내어 이름 앞에 `*`를 붙여 출력합니다.
    -   **브랜치 생성 (`branch <이름>`)**: `getHeadCommit`으로 현재 커밋 해시를 가져온 뒤, `.git/refs/heads/<새 브랜치 이름>` 파일을 만들고 그 안에 해시를 써넣으면 끝입니다. 매우 간단하죠!
