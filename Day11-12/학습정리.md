# 📚 학습 정리

## 📝 스스로 확인할 사항

### 1. Producer-Consumer 패턴과 대기 큐

- **개념**: 생산자(Producer)가 작업을 생성하여 큐(Queue)에 넣으면, 소비자(Consumer)가 큐에서 작업을 꺼내 처리하는 디자인 패턴입니다. 이 패턴은 작업 생산 속도와 소비 속도가 다를 때 시스템을 안정적으로 운영할 수 있게 해주는 핵심적인 역할을 합니다.
- **일반적인 문제와 해결 방식**:
    - **큐 오버플로우 (Queue Overflow)**: 생산 속도가 소비 속도보다 월등히 빠르면 큐에 작업이 무한정 쌓여 메모리 부족 문제가 발생할 수 있습니다. 이를 해결하기 위해 **큐의 최대 크기를 제한**하거나, 큐가 가득 찼을 때 생산자에게 압력을 가해 생산 속도를 늦추는 **역압력(Backpressure)** 메커니즘을 도입할 수 있습니다.
    - **경쟁 상태 (Race Condition)**: 여러 소비자가 동시에 큐에 접근하면 같은 작업을 중복으로 가져가거나 데이터가 손상될 수 있습니다. 이번 미션에서는 싱글 스레드 환경이라 이 문제가 없었지만, 멀티 스레드 환경이라면 **뮤텍스(Mutex)나 세마포어(Semaphore)** 같은 동기화 장치를 사용해 한 번에 하나의 스레드만 큐에 접근하도록 보장해야 합니다.
- **[구현 경험]**: 이번 미션에서 `Uploader`는 생산자, `Manager`와 `Converter`는 소비자에 해당합니다. `Manager`가 주기적으로 `Uploader`의 큐를 확인하고 `Converter`의 처리 용량(`isBusy`)을 체크하는 과정 자체가 간단한 형태의 소비자 속도 조절 메커니즘이라고 볼 수 있습니다.

### 2. 비동기 처리의 어려움: `Promise`와 `EventEmitter`

- **가장 어려운 점**: 비동기 코드의 가장 큰 어려움은 **에러 처리와 디버깅**입니다. 
    - **에러 전파의 단절**: `EventEmitter`의 리스너 콜백 함수 내부에서 발생한 에러는 `try...catch` 문으로 잡아낼 수 없습니다. 에러는 해당 이벤트 루프 틱에서 끝나버리고, 어디서 문제가 발생했는지 추적하기 어렵습니다. 이를 해결하려면 리스너 내부에 개별적으로 `try...catch`를 구현하거나, `EventEmitter`에 `error` 이벤트를 정의하여 중앙에서 처리해야 합니다.
    - **흐름 추적의 복잡성**: `Promise` 체이닝이나 `EventEmitter`의 이벤트 발생은 코드의 실행 순서와 실제 동작 순서를 다르게 만듭니다. `A -> B -> C` 순서로 코드가 작성되어도, 실제로는 `A`가 던진 이벤트에 의해 `C`가 먼저 실행되고 `B`가 나중에 실행될 수 있습니다. 이 때문에 논리적인 흐름을 머릿속으로 따라가기가 매우 어렵습니다.


### 3. 모듈 설계와 협력: 나만의 기준

- **기준 1: "묻지 말고 시켜라 "**: 모듈이 다른 모듈의 내부 상태를 묻고(ask) 그것을 기반으로 결정하는 대신, 그냥 원하는 것을 시켜야(tell) 합니다. 
    - **나쁜 예**: `Manager`가 `Converter`의 큐 길이를 직접 확인하고, `if (길이 < 2)` 같은 코드로 직접 작업을 넣는 것.
    - **좋은 예 (이번 미션의 방식)**: `Manager`는 `converter.convert(video)`라는 메시지만 보냅니다. 작업을 받을지 말지는 `Converter`가 자신의 상태(`processingCount`, `capacity`)를 보고 스스로 결정합니다. 이는 모듈의 자율성과 캡슐화를 높여줍니다.
- **기준 2: "이벤트로 말하라"**: 모듈 간의 직접적인 참조를 최소화하고, `EventEmitter`를 통해 "방송"하도록 합니다. `Converter`는 변환이 끝났을 때 `Manager`를 직접 호출하는 대신, `conversion_complete`라는 이벤트를 방송합니다. 누가 이 방송을 들을지는 `main.js`와 같은 조립기(Assembler)가 결정하므로, 모듈 간의 결합도가 낮아져 유연하고 확장 가능한 구조가 됩니다.

## 🗣️ 다같이 확인할 사항

### 1. `Promise`를 `async/await`로 변환하기

- `async/await`는 `Promise`를 더 쉽게 사용하기 위한 문법적 설탕(Syntactic Sugar)입니다. `async` 함수는 항상 `Promise`를 반환하며, `await` 키워드는 `Promise`가 처리될 때까지 함수의 실행을 일시 중지합니다.
- **추가 필요사항**:
    1.  **`async` 함수 선언**: `await`를 사용하려는 함수는 반드시 `async` 키워드로 선언되어야 합니다.
    2.  **`try...catch`를 이용한 에러 처리**: `Promise`의 `.catch()` 메소드 대신, 동기 코드처럼 `try...catch` 블록을 사용하여 `await`하는 동안 발생할 수 있는 예외(rejected Promise)를 처리해야 합니다. 이것이 `async/await`를 사용할 때 가장 중요한 부분 중 하나입니다.

    ```javascript
    // Promise.then().catch()
    somePromiseFunction()
        .then(result => { console.log(result); })
        .catch(error => { console.error(error); });

    // async/await
    async function run() {
        try {
            const result = await somePromiseFunction(); // Promise가 끝날 때까지 기다림
            console.log(result);
        } catch (error) {
            console.error(error);
        }
    }
    ```

### 2. 비동기 디버깅의 어려움과 해결책

- **어려운 점**: 비동기 코드의 가장 큰 문제는 **콜 스택(Call Stack)이 단절**된다는 점입니다. `setTimeout`이나 이벤트 리스너의 콜백 함수는 새로운 콜 스택에서 실행되므로, 에러가 발생했을 때 원래 어떤 컨텍스트에서 호출되었는지 추적하기가 매우 어렵습니다.
- **디버깅 방법 비교**:
    - **`console.log`**: 가장 원시적이지만 가장 직관적인 방법입니다. 비동기 작업의 시작, 중간, 끝 지점에 로그를 남겨 실행 흐름과 데이터 변화를 추적할 수 있습니다.
    - **Node.js `async_hooks` 모듈**: 비동기 리소스(Promise, Timeout 등)의 생명주기를 추적할 수 있는 강력한 내장 모듈입니다. 각 비동기 작업에 고유 ID를 부여하여 콜 스택이 끊어지더라도 전체 작업 흐름을 연결하여 추적할 수 있게 해줍니다. (고급 디버깅에 사용)
    - **Chrome DevTools for Node.js**: `node --inspect` 플래그를 사용하여 Node.js 애플리케이션을 실행하면, Chrome 브라우저의 개발자 도구를 연결하여 디버깅할 수 있습니다. 소스 코드에 중단점(Breakpoint)을 설정하고, 콜 스택을 시각적으로 확인하며, 비동기 작업의 흐름을 단계별로 추적할 수 있어 매우 효과적입니다.

### 3. 스레드 생성 vs 비동기 방식

| 특징 | 스레드 생성 (e.g., `worker_threads`) | 비동기 방식 (Event Loop) |
| --- | --- | --- |
| **동시성 모델** | **선점형 멀티태스킹**. OS 스케줄러가 스레드를 제어하고 CPU 시간을 할당. | **협력적 멀티태스킹**. 이벤트 루프가 단일 스레드에서 작업을 순환하며, 각 작업이 자발적으로 제어권을 넘겨야 함. |
| **자원 사용량** | 스레드마다 별도의 콜 스택과 메모리 공간이 필요하여 **자원 소모가 큼**. | 단일 스레드 기반으로 **메모리 사용량이 매우 적고** 가벼움. |
| **컨텍스트 스위칭** | 스레드 간 전환 시 **비용이 많이 듦** (CPU 캐시 무효화 등). | 이벤트 루프 내에서의 작업 전환은 함수 호출에 가까워 **매우 빠름**. |
| **적합한 작업** | **CPU 집약적인 작업** (예: 복잡한 수학 연산, 암호화, 비디오 인코딩). | **I/O 집약적인 작업** (예: 파일 읽기/쓰기, 네트워크 요청, 데이터베이스 쿼리). |
| **제약사항** | 스레드 개수가 늘어날수록 컨텍스트 스위칭 비용이 증가하여 성능이 저하될 수 있음. 공유 자원에 대한 **동기화 문제(데드락 등)를 직접 처리**해야 함. | **CPU 집약적인 작업에 취약**. 하나의 작업이 이벤트 루프를 오래 점유하면 전체 시스템이 멈추는 **"블로킹"** 현상 발생. |
