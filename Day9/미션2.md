# **학습 목표**

- 멀티 스레드에서 이벤트 기반 동작을 도와주는 비동기 이벤트 매니저를 직접 구현할 수 있다.
- Publisher-Subscriber 패턴을 사용하는 경우와 어떤 구현 방식이 있는지 학습할 수 있다.
- Publisher와 Subscriber 역할을 도와줄 수 있는 EventManager 객체를 직접 구현할 수 있다.
- 동시성 흐름 제어를 위해 EventEmitter 라이브러리를 학습하고, 비동기 방식으로 동작하도록 구현한다.

# **사전지식**

비동기 대기큐 방식 학습이 필요합니다.

# **기능요구사항**

다음과 같이 동작하는 비동기 방식을 지원하는 EventManager 클래스를 구현하고, 동작을 확인하기 위해서 호출하는 프로그램도 작성한다.

- subscriber도 handler를 처리하는 Event Emitter를 지정할 수 있도록 개선한다.
- handler에 Event 와 함께 이벤트가 종료를 의미하는 **`completed`** flag를 전달한다.

### **메소드**

- **`sharedInstance()`** 싱글톤으로 접근하도록 함수를 구현한다. 여러 번 호출해도 한 번 생성한 동일한 인스턴스를 반환한다.
- **`add(subscriber, eventName, sender, handler)`** Subscriber를 추가할 때는 subscriber 객체와 이벤트 이름, 이벤트 발행 객체를 모두 합쳐서 고유한 키로 봐야 한다.
    - subscriber 와 sender 모두 객체 인스턴스를 인자값으로 전달한다.
    - eventName 은 이벤트를 지칭하는 이름이다. "" 빈값도 가능하다.
    - handler는 클로저로 이벤트를 받을 때 실행되어야 한다. 매개변수로 이벤트를 전달받는다.
    - Subscriber를 등록할 때 Event Emitter를 지정할 수 있다.
    - handler에는 completed 플래그를 추가하고, subscriber에서 completed에 따라 다르게 동작할 수 있도록 한다.
- **`remove(subscriber)`**
    
    subscriber를 제거할 때는 subscriber로 등록된 모든 조건을 제거한다.
    
- **`postEvent(eventName, sender, userData)`**
    - 이벤트를 발행할 때는 이벤트 이름을 꼭 명시해야 한다.
    - 이벤트를 발생하고 전송하는 Publisher 객체 인스턴스를 꼭 넘겨야 한다.
    - Dictionary<String, String>타입으로 userData를 전달할 수 있다. **`nil`**로 생략 가능하다.
    - 내부에서 Event 객체를 생성하고 구독 핸들러에 전달한다.
    - 이벤트를 발생하는 postEvent() 메소드를 sync 방식, async 방식, delay 방식을 지원하는 메소드를 구분해서 구현한다.
    - completed 플래그도 추가해서 해당 이벤트가 완료된 것인지 아닌지를 함께 전달한다.
        - 만약 동일한 이벤트가 여러번 발생하다가 끝났다는 것을 나타낼 때 사용한다.
- **`description()`**
    - 모든 Subscriber 조건을 표시하는 문자열을 리턴한다.

# **프로그래밍 요구사항**

- 반드시 Event Emitter를 적용해서 구현해야 한다
- EventManager 구현한 파일과 별도로, 위의 동작 예시처럼 이벤트 이름과 발행 객체 조건을 모두 확인할 수 있도록 실행하는 main 프로그램을 작성한다.
- 모든 Subscriber 조건과 실행 결과도 함께 출력해서 gist에 저장한다.
- 모든 Publisher 타입도 description() 함수로 자신을 설명할 수 있어야 한다.

![event-manager](https://lucas-image.codesquad.kr/1656053347325event-manager.png)

- EventManager 내부에는 그림처럼 이벤트 조건을 비교할 수 있는 규칙 데이터가 있어야 한다. 이벤트가 발생했을 때 Subscriber 구독자를 찾기 위한 데이터 구조와 표를 readme에 추가한다
- 동기방식은 postEvent()는 모든 subscriber에 이벤트 브로드캐스트 동작이 끝나고 나서 리턴한다.
- 비동기방식 postEvent()는 특정한 Event Emitter에 코드를 넣고, 곧바로 리턴한다.
- 지연 혹은 예약 방식 postEvent()는 비동기 방식처럼 특정한 Event Emitter에 코드를 넣고, 지정한 시각에 해당 코드가 실행되야 한다.
- Subscriber 핸들러를 실행하는 Event Emitter가 어떤 큐인지 확인해서 출력한다. 특히 main 스레드용 Queue인지 아닌지 확인이 필요하다.
- 비동기방식이나 지연 방식을 테스트하려면 main()에서 기다리는 코드가 필요하다.
- 비동기와 지연 방식을 확인하기 위해서 이벤트 전송 시각과 subscriber 코드 동작 시각을 출력한다.
- 모든 Subscriber 조건과 실행 결과도 함께 출력해서 gist에 저장한다.
- 출력 형식은 자신만의 방식으로 출력한다.

- A는 albumModel 객체가 전송한 "ModelDataChanged" 이벤트만 받는다.
- B는 이벤트 이름이 빈 문자열이기 때문에 albumItemView 객체가 전송하는 모든 이벤트를 받는다.
- C는 albumController가 전송한 "DidShakeMotion" 이벤트를 받는다.
- C는 sender가 null로 비어있기 때문에 모든 객체가 보내는 "AllDataChanged" 이벤트는 모두 받는다.
- D는 sender가 null이고, 이벤트 이름도 빈 문자열이기 때문에 모든 객체가 보내는 모든 이벤트를 다 받는다.

- subscriberA는 albumModel 객체가 전송한 "ModelDataChanged" 이벤트를 XXX emitter에서 처리하고 싶어한다.
