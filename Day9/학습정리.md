# 📚 Day 9: 학습 정리

## 1. 프로세스와 스레드, 그리고 동시성

### 1.1 프로세스 vs 스레드

- **프로세스(Process)란?**
    - 운영체제로부터 자원을 할당받는 작업의 단위. 자신만의 독립된 메모리 영역(Code, Data, Heap, Stack)을 가집니다.
- **스레드(Thread)란?**
    - 프로세스 내에서 실행되는 흐름의 단위. Stack을 제외한 Code, Data, Heap 영역을 다른 스레드와 **공유**합니다.
- **가장 큰 차이점은?**
    - **메모리 공유 여부**. 프로세스 간 통신(IPC)은 복잡하지만, 스레드 간 통신은 공유 메모리를 통해 비교적 쉽게 이루어집니다. 이로 인해 스레드 간 협력이 효율적이지만, 동기화 문제가 발생할 수 있습니다.

### 1.2 경쟁 상태 (Race Condition)

- **경쟁 상태는 왜 발생하는가? (Read-Modify-Write 문제)**
    1.  **읽기(Read):** 스레드 A가 공유 변수 `count` (값: 10)를 읽음.
    2.  **컨텍스트 스위칭(Context Switching):** 운영체제가 스레드 A의 실행을 잠시 멈추고, 스레드 B에게 CPU 제어권을 넘김.
    3.  **읽기(Read):** 스레드 B가 공유 변수 `count` (값: 10)를 읽음.
    4.  **수정(Modify) 및 쓰기(Write):** 스레드 B가 `count + 1` (11)을 계산하고, 공유 변수 `count`에 11을 씀.
    5.  **컨텍스트 스위칭:** 다시 스레드 A가 실행됨.
    6.  **수정(Modify) 및 쓰기(Write):** 스레드 A는 이전에 읽었던 값 10을 기준으로 `count + 1` (11)을 계산하고, 공유 변수 `count`에 11을 씀. (스레드 B의 작업이 무시됨)
- **결과:** 두 스레드가 작업을 수행했지만, `count`는 12가 아닌 11이 됨. 이처럼 실행 순서에 따라 결과가 달라지는 현상이 경쟁 상태입니다.

### 1.3 동기화 기법 (Synchronization)

- **뮤텍스(Mutex)와 세마포어(Semaphore):**
    - **임계 구역(Critical Section):** 공유 자원에 접근하는 코드 영역을 의미. 이 영역은 한 번에 하나의 스레드만 접근 가능해야 합니다.
    - **뮤텍스:** 하나의 스레드만 임계 구역에 들어갈 수 있도록 하는 잠금(Lock) 메커니즘. 화장실이 1개뿐이라 키를 가진 사람만 들어갈 수 있는 것과 유사합니다.
    - **세마포어:** 정해진 개수(N)의 스레드만 임계 구역에 들어갈 수 있도록 하는 메커니즘. N개의 칸이 있는 주차장과 유사합니다.

### 1.4 JavaScript의 동기화: `Atomics`와 `SharedArrayBuffer`

- **`SharedArrayBuffer`란?**
    - 일반적인 `postMessage`가 데이터를 복사해서 전달하는 것과 달리, 여러 Worker 스레드가 **직접 공유할 수 있는 메모리 영역**을 만들어주는 객체입니다. 진정한 의미의 공유 메모리 구현의 핵심입니다.
- **`Atomics`란?**
    - `SharedArrayBuffer`로 만들어진 공유 메모리의 값을 **원자적으로(atomic)** 읽고, 쓰고, 수정할 수 있게 해주는 도구입니다. `Atomics.add()`, `Atomics.load()`, `Atomics.store()` 등의 메서드를 통해 Read-Modify-Write 문제를 해결하고 경쟁 상태를 방지합니다. OS 수준의 뮤텍스/세마포어 역할을 JS 환경에서 수행합니다.
- **`Atomics`는 어떻게 경쟁 상태를 해결하는가?**
    - `Atomics.add(buffer, index, value)`와 같은 연산은 CPU 수준에서 단일 명령으로 처리되도록 보장됩니다. 즉, 값을 읽고, 더하고, 쓰는 과정이 중간에 다른 스레드에 의해 중단되지 않아 데이터의 일관성을 유지할 수 있습니다.

## 2. 비동기 프로그래밍과 이벤트 기반 아키텍처

### 2.1 Publisher-Subscriber 패턴

- **이 패턴은 왜 사용하는가?**
    - 컴포넌트 간의 **느슨한 결합(Loose Coupling)**을 위함입니다. Publisher(발행자)는 Subscriber(구독자)가 누구인지, 무엇을 하는지 알 필요 없이 오직 메시지(이벤트)만 발행하면 됩니다. 이를 통해 시스템의 유연성과 확장성을 높일 수 있습니다.
- **`EventManager`의 역할:**
    - Publisher와 Subscriber 사이를 중재하는 중앙 관리자(Broker) 역할을 합니다. 이벤트 발행 및 구독 요청을 받아, 조건에 맞는 구독자에게 이벤트를 전달해 줍니다.

### 2.2 Node.js의 이벤트 루프와 비동기 처리

- **이벤트 루프(Event Loop)란?**
    - Node.js가 싱글 스레드임에도 불구하고 비동기 작업을 효율적으로 처리할 수 있게 하는 핵심 메커니즘입니다. Call Stack, Task Queue, Microtask Queue 등을 감시하며, Call Stack이 비었을 때 Queue에 있는 작업(콜백 함수)을 가져와 실행합니다.
- **`setTimeout(..., 0)` vs `process.nextTick()`**
    - 둘 다 "지금 말고 다음 순서에" 실행하라는 의미지만, 실행 시점이 다릅니다. `nextTick`은 현재 실행 중인 작업이 끝나자마자 이벤트 루프의 다음 틱(tick)으로 넘어가기 **전에** 실행(Microtask Queue)됩니다. `setTimeout(..., 0)`은 이벤트 루프의 다음 틱 **이후에** 실행(Task Queue)됩니다. 일반적으로 `nextTick`이 더 먼저 실행됩니다.
- **이벤트 루프 큐의 우선순위:**
    - 이벤트 루프는 항상 **마이크로태스크 큐(Microtask Queue)를 먼저 확인**하고, 그 안에 있는 모든 작업을 처리한 뒤에 **태스크 큐(Task Queue)에서 작업을 하나 가져와 처리**합니다. 따라서 `process.nextTick`이나 `Promise`의 콜백이 `setTimeout`의 콜백보다 항상 먼저 실행됩니다.
    - **실행 순서 예측 예제:**
      ```javascript
      console.log('1. 동기 코드 시작');
      setTimeout(() => console.log('2. setTimeout (Task Queue)'), 0);
      Promise.resolve().then(() => console.log('3. Promise (Microtask Queue)'));
      process.nextTick(() => console.log('4. nextTick (Microtask Queue)'));
      console.log('5. 동기 코드 종료');
      // 예상 결과: 1 -> 5 -> 4 -> 3 -> 2
      ```

### 2.3 `EventManager`의 구독자 검색 로직

- **이벤트 발생 시, 어떤 구독자에게 전달해야 하는가?**
    - `postEvent(eventName, sender)`가 호출되면, `EventManager`는 등록된 모든 구독자 목록을 순회하며 아래 조건을 모두 만족하는 구독자를 찾습니다.

| 구독자 조건 (`sub`) | 발행된 이벤트 (`post`) | 전달 여부 | 설명 |
| :--- | :--- | :--- | :--- |
| `sub.eventName` == `post.eventName` | `sub.sender` == `post.sender` | **O** | 이벤트 이름과 발행자가 모두 일치 (가장 구체적인 조건) |
| `sub.eventName` == `""` | `sub.sender` == `post.sender` | **O** | 특정 발행자가 보내는 모든 이벤트를 구독 |
| `sub.eventName` == `post.eventName` | `sub.sender` == `null` | **O** | 누가 보내든 특정 이름의 이벤트를 모두 구독 |
| `sub.eventName` == `""` | `sub.sender` == `null` | **O** | 모든 발행자의 모든 이벤트를 구독 (가장 일반적인 조건) |
| (그 외) | (그 외) | **X** | 조건 불일치 |

### 2.4 `EventManager` 성능 개선 고려사항

- **배열 순회의 한계:**
    - 구독자가 수천, 수만 개로 늘어날 경우, `postEvent`가 호출될 때마다 전체 배열을 순회(`filter`, `forEach`)하는 방식은 성능 저하를 유발할 수 있습니다. 이벤트 하나를 발행하기 위해 불필요한 검색을 많이 해야 합니다.
- **Map을 이용한 성능 개선:**
    - `Map` 객체를 사용하여 `eventName`을 Key로, 해당 이벤트를 구독하는 구독자 리스트를 Value로 저장하면 검색 효율을 높일 수 있습니다. `subscribers.get(eventName)`을 통해 필요한 구독자 목록에 바로 접근할 수 있어, 전체를 순회할 필요가 없어집니다.

## 3. 회고 및 심화 학습 (지도)

### 3.1 Worker Thread 동작 방식

Node.js의 `worker_threads`는 단순한 비동기 작업이 아닌, OS 수준의 실제 스레드를 생성하여 병렬 처리를 가능하게 합니다. 각 워커 스레드는 메인 스레드와 독립된 V8 엔진 인스턴스와 이벤트 루프를 가지며, 자신만의 격리된 환경에서 코드를 실행합니다. 

- **통신:** 메인 스레드와 워커는 `postMessage`와 `on('message')`를 통해 메시지를 주고받습니다. 이 과정에서 데이터는 직접 공유되는 것이 아니라, 구조화된 복제 알고리즘(Structured Clone Algorithm)을 통해 복사됩니다. 이는 데이터의 안전성을 보장하지만, 큰 데이터를 주고받을 때는 성능 저하의 원인이 될 수 있습니다.
- **진정한 공유:** `SharedArrayBuffer`는 이 규칙의 유일한 예외입니다. 이 객체를 통해 여러 스레드가 실제 메모리 공간을 공유할 수 있으며, 이를 통해 고성능 병렬 연산이 가능해집니다. 하지만 이는 곧 경쟁 상태의 위험에 노출된다는 의미이므로, `Atomics`를 통한 동기화 제어가 반드시 필요합니다.

### 3.2 Promise와 Event Emitter 도입의 어려움

- **Promise의 어려움:**
    - **에러 추적의 복잡성:** `.then()` 체인이 길어질수록 어디서 에러가 발생했는지 추적하기 어려울 수 있습니다. 모든 `Promise` 체인 끝에 `.catch()`를 붙이거나, `unhandledRejection` 이벤트를 처리하지 않으면 에러가 조용히 무시될 수 있습니다.
    - **흐름 제어:** 여러 개의 비동기 작업을 동시에 시작하고 모두 끝날 때까지 기다리는(`Promise.all`), 혹은 순차적으로 실행하는 등의 흐름을 제어하기 위해서는 `Promise` API에 대한 명확한 이해가 필요합니다.
- **Event Emitter의 어려움:**
    - **상태 부재:** `EventEmitter`는 단지 이벤트가 발생했다는 사실만 알릴 뿐, 현재의 '상태'를 저장하지 않습니다. 따라서 중요한 일회성 이벤트가 발생한 후에 구독한 리스너는 해당 이벤트를 영원히 놓치게 됩니다.
    - **메모리 누수:** 이벤트 리스너를 등록한 후, 더 이상 필요 없을 때 `removeListener`로 명시적으로 제거해주지 않으면 해당 리스너와 관련된 객체들이 메모리에 계속 남아 누수를 일으킬 수 있습니다.

### 3.3 Event Emitter와 비동기 동작

기본적으로 Node.js의 `EventEmitter`는 **동기적으로 동작**합니다. `emitter.emit('event')`를 호출하면, 등록된 리스너들이 즉시, 그리고 순차적으로 호출됩니다. 

`EventManager` 미션에서 `postEventAsync`를 구현했듯이, `EventEmitter`를 비동기적으로 동작시키려면 리스너 실행 부분을 `setTimeout`, `setImmediate`, `process.nextTick` 등으로 감싸서 이벤트 루프의 다음 사이클로 실행을 위임해야 합니다. 이를 통해 이벤트 발행 코드가 리스너의 실행을 기다리지 않고 즉시 다음 작업을 수행할 수 있게 됩니다.

### 3.4 Promise와 async/await

`async/await`는 Promise를 더 쉽게 사용하기 위한 **문법적 설탕(Syntactic Sugar)**입니다. `async` 키워드가 붙은 함수는 항상 `Promise`를 반환하며, `await` 키워드는 `Promise`가 완료(settled)될 때까지 함수의 실행을 일시 중지하고 기다립니다.

Promise 기반의 함수를 `async/await`로 처리하기 위해 추가로 필요한 사항은 다음과 같습니다.

1.  **`async` 함수 컨텍스트:** `await`는 반드시 `async`로 선언된 함수 내부에서만 사용할 수 있습니다.
2.  **`try...catch`를 이용한 에러 처리:** Promise의 `.catch()` 체인 대신, 동기 코드처럼 `try...catch` 구문을 사용하여 `await`하는 동안 발생할 수 있는 예외(Promise의 rejection)를 처리해야 합니다.

```javascript
// Promise.then/.catch
myPromiseFunc()
  .then(result => { console.log(result); })
  .catch(error => { console.error(error); });

// async/await
async function run() {
  try {
    const result = await myPromiseFunc();
    console.log(result);
  } catch (error) {
    console.error(error);
  }
}
```

### 3.5 Observer 패턴 vs Publisher-Subscriber 패턴

두 패턴 모두 객체 간의 느슨한 결합을 통해 이벤트 기반 아키텍처를 구현하는 것을 목표로 하지만, 중요한 차이점이 있습니다.

| 구분 | Observer 패턴 | Publisher-Subscriber 패턴 |
| :--- | :--- | :--- |
| **결합도** | Subject(발행자)가 Observer(구독자)를 직접 알고 목록으로 관리. **상대적으로 결합도가 높음.** | Publisher와 Subscriber가 서로를 전혀 모름. **중앙의 Broker(이벤트 채널)를 통해 통신.** | 
| **중개자** | 없음. Subject가 직접 Observer에게 알림. | **Broker(Event Bus)가 존재.** Publisher는 Broker에게, Broker가 Subscriber에게 전달. | 
| **통신 방식** | 주로 동기적으로 직접 메소드를 호출. | Broker의 구현에 따라 동기/비동기 모두 가능. **주로 비동기 방식**에 사용됨. | 
| **예시** | 데이터 모델(Subject)이 변경될 때, 그 데이터를 보여주는 여러 UI 뷰(Observer)를 업데이트하는 경우. | 채팅 애플리케이션에서 여러 사용자가 특정 채팅방(채널)에 메시지를 보내고 받는 경우. | 

이번 미션에서 구현한 `EventManager`는 Publisher와 Subscriber 사이에 존재하는 **Broker** 역할을 하므로, **Publisher-Subscriber 패턴**에 해당합니다.
