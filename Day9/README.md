# Day 9: 멀티 스레딩, 동시성 제어와 비동기 이벤트 매니저

## 🚀 미션 목표

이번 미션은 현대 멀티코어 환경의 핵심인 **동시성 프로그래밍**을 깊이 있게 이해하는 것을 목표로 합니다. 먼저 Node.js의 `worker_threads`를 사용하여 멀티 스레딩 환경을 직접 구축하고, 공유 자원 접근 시 발생하는 **경쟁 상태(Race Condition)** 문제를 의도적으로 재현하고 분석합니다. 

이후, 이러한 동시성 문제를 효과적으로 관리하고 해결하기 위한 아키텍처 패턴으로, **Publisher-Subscriber 패턴** 기반의 비동기 이벤트 매니저(Async Event Manager)를 직접 구현합니다. 이를 통해 동기/비동기/지연 처리가 가능한 정교한 이벤트 시스템을 설계하고, 복잡한 동시성 흐름을 안전하고 체계적으로 제어하는 방법을 학습합니다.

## 📝 프로그래밍 요구사항

### 미션 1: 경쟁 상태(Race Condition) 재현

- **멀티 스레드 환경 구축:** Node.js의 `worker_threads` 모듈을 사용하여 여러 개의 스레드를 생성합니다.
- **공유 자원 접근:** 모든 스레드가 하나의 공유된 변수(또는 메모리)에 반복적으로 읽기(read)와 쓰기(write) 작업을 수행하도록 구현합니다.
- **경쟁 상태 유발:** 각 스레드의 작업 사이에 미세한 시간 차이(`setTimeout` 등)를 두어, 운영체제의 스레드 스케줄링에 따라 실행 순서가 뒤바뀌고 데이터의 최종 일관성이 깨지는 경쟁 상태를 의도적으로 발생시킵니다.
- **동작 검증:** 어떤 스레드가 언제, 어떤 값을 읽고 썼는지 로그를 통해 명확히 확인할 수 있는 검증 로직을 포함합니다.
- **스레드 제어:** 스레드 생성, 데이터 전달(main ↔ worker), 스레드 종료 등 기본적인 스레드 생명주기 관리 API를 활용합니다.

### 미션 2: 비동기 이벤트 매니저 `EventManager` 구현

- **싱글톤 패턴:** `sharedInstance()` 메소드를 통해 항상 동일한 `EventManager` 인스턴스를 반환하도록 구현합니다.
- **Pub/Sub 모델 구현:**
    - `add(subscriber, eventName, sender, handler)`: 특정 조건(이벤트 이름, 발행 객체)에 따라 이벤트를 구독(subscribe)하고, 실행할 핸들러(클로저)를 등록합니다.
    - `remove(subscriber)`: 특정 구독자의 모든 구독 정보를 제거합니다.
    - `postEvent(...)`: 이벤트를 발행(publish)합니다.
- **다양한 발행 방식 지원:**
    - **동기(Sync):** `postEvent` 호출 시, 모든 구독자의 핸들러 실행이 끝나야 반환됩니다.
    - **비동기(Async):** `postEvent` 호출 시, 핸들러 실행을 별도의 이벤트 큐(Event Emitter)에 위임하고 즉시 반환됩니다.
    - **지연(Delay):** 지정된 시간 후에 비동기적으로 핸들러가 실행됩니다.
- **정교한 이벤트 처리:**
    - `completed` 플래그를 핸들러에 전달하여, 연속된 이벤트의 마지막을 알릴 수 있도록 합니다.
    - `userData`를 통해 이벤트 발행 시 부가적인 데이터를 전달할 수 있도록 합니다.
- **상태 확인:** `description()` 메소드를 통해 현재 등록된 모든 구독자 정보를 문자열로 반환합니다.

## 💻 구현 과정

### 미션 1: 경쟁 상태(Race Condition) 재현

1.  **기본 스레드 통신 구현:** `worker_threads` 모듈을 사용하여 메인 스레드와 워커 스레드를 생성하고, `postMessage`와 `on('message')`를 통해 서로 데이터를 주고받는 가장 기본적인 통신 구조를 구현하며 동작 방식을 익혔습니다.
2.  **공유 메모리 도입:** 단순 데이터 복사 방식인 `postMessage`로는 경쟁 상태를 재현하기 어렵다고 판단하여, 실제 메모리 주소를 공유하는 `SharedArrayBuffer`를 도입했습니다. `Int32Array`를 사용하여 해당 버퍼를 정수 배열처럼 다룰 수 있도록 했습니다.
3.  **경쟁 상태 유발 로직 구현:** 각 워커가 공유 메모리의 값을 읽고(Read), `setTimeout`을 이용해 의도적으로 짧은 지연을 준 뒤, 1을 더한 값을 다시 쓰는(Write) 'Read-Modify-Write' 로직을 구현했습니다. 이 지연 시간 동안 다른 스레드가 끼어들면서 경쟁 상태가 발생하는 것을 유도했습니다.
4.  **결과 검증:** 모든 워커의 작업이 종료된 후, 공유된 최종 결과값을 출력했습니다. 5개의 스레드가 10번씩 총 50번을 더했음에도 불구하고, 최종 결과는 10 근처의 작은 값으로 나오며 경쟁 상태가 성공적으로 재현되었음을 확인했습니다.

### 미션 2: 비동기 이벤트 매니저 `EventManager` 구현

1.  **싱글톤 구조 설계:** `EventManager`가 애플리케이션 전역에서 단 하나의 인스턴스로 관리되어야 하므로, `static` 필드와 `sharedInstance()` 메소드를 이용한 싱글톤 패턴으로 기본 구조를 설계했습니다.
2.  **핵심 기능 구현:** 구독자 정보를 담을 내부 배열(`subscribers`)을 만들고, `add`, `remove`, `description` 메소드를 구현했습니다. `remove` 시에는 `filter`를 사용하여 특정 구독자의 모든 구독 정보를 한 번에 제거하는 로직을 구현했습니다.
3.  **동기 발행 구현:** `postEvent` 메소드를 구현하여 조건(이벤트 이름, 발행 객체)에 맞는 구독자를 필터링하고, `forEach`로 순회하며 핸들러를 즉시 실행하는 동기 방식의 이벤트 발행 기능을 먼저 완성했습니다.
4.  **비동기/지연 발행 확장:** `setTimeout`을 활용하여 핸들러 실행을 이벤트 루프의 다음 틱으로 위임하는 `postEventAsync`와, 지정된 시간만큼 지연시키는 `postEventWithDelay`를 구현했습니다. 이 과정에서 `completed` 플래그와 `userData`를 이벤트 객체로 묶어 전달하는 요구사항도 반영했습니다.
5.  **단계별 테스트:** 각 기능을 구현할 때마다 별도의 테스트 파일(`event-manager-test.js`)을 작성하고 실행하여, 싱글톤, 구독 관리, 동기/비동기/지연 발행 등 모든 기능이 의도대로 동작하는지 단계적으로 검증하며 진행했습니다.

## 🔧 트러블슈팅

-   **문제 상황:** 비동기(`postEventAsync`) 및 지연(`postEventWithDelay`) 발행 기능을 테스트할 때, 메인 테스트 스크립트가 비동기 작업의 완료를 기다리지 않고 먼저 종료되어 이벤트 처리 로그를 확인할 수 없는 문제가 있었습니다.
-   **해결 과정:** 이는 Node.js의 이벤트 루프가 동작할 시간을 주지 않고 메인 스레드가 먼저 끝나버리기 때문에 발생하는 문제였습니다. 이를 해결하기 위해, 테스트 스크립트의 마지막에 모든 비동기 작업이 완료되기에 충분한 시간(예: 1000ms)을 기다리는 `setTimeout`을 추가했습니다. 이를 통해 메인 프로세스가 살아있는 동안 지연되었던 이벤트 핸들러들이 정상적으로 실행되고 로그가 출력되는 것을 확인할 수 있었습니다.
