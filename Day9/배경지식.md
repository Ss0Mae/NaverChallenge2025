# **동기와 비동기 작업**

동기 작업, 비동기 작업 처리 방식에 대해 비교하고 학습한다.

### **𝑰. 동기 작업으로 이뤄지는 함수 호출 과정**

![sync-call](https://lucas-image.codesquad.kr/1671524226068sync-call.gif)

- 메인 흐름에서 **`area()`**함수를 호출 하면 call stack에 area()가 쌓인다.
- area() 내부 코드에서 **`distance()`** 함수를 호출하고 나서 return 될 때까지 멈춘다 (blocking)
- distance()가 호출되면 call stack에 distance()가 area() 위에 쌓인다
- distance()가 리턴되면 call stack에서 distance()가 사라지고, 멈춰있던 area() 다음 코드가 실행된다
- area() 내부 코드가 모두 실행되고 나면 area()도 call stack에서 사라진다
    
    예전 방식으로 설명하자면, 이렇게 코드가 한 줄씩 절차적으로 실행되는 과정을 Routine이라고 하며, 상위에서 하위에 다른 함수를 호출하면 Subroutine을 실행한다.
    
    요즘은 루틴-서브루틴을 구분하지 않지만 하위 함수를 호출하는 **`caller`**와 호출되서 실행하는 **`callee`**는 구분한다.
    

> 여기서 루틴 동작 설명은 작동하는 일꾼 worker가 한 명만 있다는 가정으로 설명했다. 프로그램 실행 관점에서 일꾼은 스레드와 일치한다고 봐도 무리가 없다.
> 

### **𝑰𝑰. 비동기 함수 호출 과정**

![async-callback](https://lucas-image.codesquad.kr/1671530371529async-callback.gif)

동기 작업과 다르게 비동기 작업으로 호출하는 과정은 호출하는 대상 **`context#1`**과 작업을 완료하고 결과를 호출하는 **`context#2`**가 분리되어서 동작 주체가 서로 다르다.

- **`caller`**는 context#2가 실행할 함수를 호출하지만, 거의 즉시 리턴되서 블록되서 기다리지 않는다
- **`callee`**는 동작 결과를 알려주기 위해서 콜백함수(포인터)를 받거나 클로저를 전달받는다
- context#1에서는 호출 시점에 잠시 블록이 되지만 곧바로 리턴되기 때문에 계속해서 다음 줄을 실행한다
- context#2에서는 caller가 요청한 작업을 시작한다. 파일을 읽는다거나 타이머를 시작하거나 코루틴을 실행한다. 요청한 작업이 끝나면 context#2에서 콜백함수(또는 클로저)를 실행한다.

> 여기서 context#1과 context#2는 각각 다른 스레드라고 이해해도 무방하다.
> 

### **𝑰𝑰𝑰. 동기화**

비동기 작업은 서로 다른 스레드에서 실행하다보니 다른 작업이 끝나길 기다리거나, 동시에 접근하지 않도록 동기화 방식이 필요하다

![race-condition](https://lucas-image.codesquad.kr/1671532067400race-condition.png)

서로 다른 스레드에서 실행되는 코드가 공유하는 프로세스 메모리에 접근할 때 생기는 문제를 **`Race condition`**이라고 부른다.

![critical-section](https://lucas-image.codesquad.kr/1671532082134critical-section.png)

멀티 프로세스 사이에서 공유하는 리소스에서도 발생할 수 있으며, 동일한 이슈가 멀티 스레드 사이에서도 발행할 수 있다. 세마포어, 뮤텍스, 임계 영역처럼 atomic한 접근을 보장해주는 운영체제 기능으로 동시 접근을 막도록 보호해야 한다.

### **교착 상태 Dead-Lock**

아쉽게도 임계 영역을 Lock해서 보호해준다고 해서 모든 문제가 해결되지는 않는다. 아래 그림은 교착 상태의 대표적인 예제인 **`승려 문제`**다. 동그란 식탁에 승려들이 5명 앉아 있고 젓가락이 양쪽에 하나씩만 있어서 두 개를 모아야 식사할 수 있는 조건이다.

![dead-lock](https://lucas-image.codesquad.kr/1671532941685dead-lock.png)

이 경우 모두가 식사하고 기도하려면 (공유하는 리소스 = 젓가락)에 대해 동기화 문제가 있어서 다른 승려를 기다려야 하는 상황이 생기는 데, 이 때 서로가 서로를 기다리는 교착 상태가 생긴다. 프로세스나 스레드도 교착 상태가 발생하면 멈춰버릴 수 있다. 이런 상황을 피하거나 타임 아웃을 두고 교체해줘야 한다.

### **𝑰𝑽. 학습거리**

- 멀티 스레드 방식과 스레드에서 비동기 병렬 처리 방식에 대해 학습하고 정리한다.
- 특정 스레드에서 이벤트 루프를 만들어서 각 이벤트를 전달하는 방식에 대해 학습한다.
- 비동기 작업을 동기화해서 기다리는 방식에 대해 학습한다.
- 비동기 작업을 그룹으로 묶어서 동기화하는 방식에 대해 학습한다.
- 노드 환경에서 이벤트 루프와 이벤트 처리(Event Emitter) 방식에 대해 학습하고 정리한다.
- 자바는 CompletableFuture 방식에 대해 학습하고 정리한다.
- 코틀린은 코루틴을 활용하는 방식에 대해 학습하고 정리한다.
- 스위프트는 OperationQueue 방식에 대해 학습하고 정리한다.
- 여러 Thread를 미리 만들어서 Pool 형태로 관리하는 방식에 대해 학습한다.
