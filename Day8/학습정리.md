# 📚 Day 8: 학습 정리

## 1. 함수형 프로그래밍 핵심 개념

### 1.1 불변성 (Immutability)

- **불변성이란 무엇인가?**
  - 데이터가 생성된 후에 그 상태를 변경할 수 없는 특성. 변경이 필요할 경우, 원본을 바꾸는 대신 변경된 내용을 반영한 새로운 데이터를 생성한다.
- **불변성을 지키면 어떤 장점이 있는가?**
  - **예측 가능성:** 데이터가 어디서 어떻게 바뀌는지를 추적할 필요가 없고, 데이터의 변경은 항상 명시적인 새로운 값의 생성으로만 이루어지므로 버그를 줄이고 디버깅이 쉬워진다.
  - **동시성 제어:** 여러 스레드가 같은 데이터를 접근하더라도 데이터 자체는 변경되지 않으므로 Lock 없이도 안전하게 동시 접근이 가능하다.
  - **상태 관리의 단순화:** React나 Redux에서 상태를 변경할 때 불변성을 강조하는 이유는, 이전 상태와 새로운 상태를 비교(diffing)하기 쉽고, UI 업데이트 조건을 효율적으로 파악할 수 있기 때문이다.

### 1.2 순수 함수 (Pure Function)와 참조 투명성 (Referential Transparency)

- **순수 함수의 두 가지 조건은 무엇인가?**
  1. 동일한 입력에 대해 항상 동일한 출력을 반환한다
  2. 함수의 실행이 외부 상태에 영향을 주지 않는다 (No Side Effects).
- **부작용(Side Effect)이란 무엇이며, 어떤 종류가 있는가?**
  - 함수 외부의 상태를 변경하거나, 외부 세계에 영향을 미치는 모든 동작
  - 예: 전역 변수 수정, 콘솔 출력, 파일 입출력, 네트워크 요청 등
- **참조 투명성이란 무엇인가?**
  - 어떤 표현식을 그것의 결과값으로 치환해도 프로그램의 의미나 동작이 변하지 않는 특성. 순수 함수는 항상 참조 투명성을 가진다.

## 2. 불변 데이터 구조 설계 및 구현

### 2.1 불변 연결 리스트 (Immutable Linked List)

- **데이터 변경 시 새로운 인스턴스를 반환하는 원리:**

  - 기존 노드들을 그대로 재사용하고, 변경된 노드와 그 이후 노드만 새로 생성함
  - 예: 3 → 4 → 5 에서 4를 9로 변경하면
    새로운 리스트: 3 → 9 → 5 (3은 재사용, 9와 그 이후 노드는 새로 생성됨)
    → 이를 구조적 공유(Structural Sharing) 또는 Path Copying이라고 한다

- **반복문 대신 재귀를 사용하는 이유와 장단점:**
  - **장점:** 상태를 별도 변수로 추적할 필요 없이, 함수 호출 스택으로 로직을 자연스럽게 표현할 수 있음
    (→ 선언적, 간결, 직관적)
  - **단점:** 깊은 재귀 호출 시 콜 스택 초과(Stack Overflow) 위험이 있음
    → 이를 해결하기 위한 방법이 **TCO(Tail Call Optimization)**이지만, 자바스크립트는 대부분의 엔진에서 이를 지원하지 않음

### 2.2 불변 해시맵 (Immutable HashMap)

- **해시 충돌(Hash Collision) 해결 전략:**
  - 이번 미션에서는 Separate Chaining 방식 사용

  - 같은 버킷에 여러 key가 해시 충돌로 들어올 경우, 불변 연결 리스트로 관리


- **`put` 동작 시 불변성을 유지하는 과정:**
  1.  새로운 key의 해시값을 계산하여 bucket index 결정
  2.  해당 index의 연결 리스트를 복사하여 key-value 삽입
  3.  전체 bucket 배열도 shallow copy한 후 해당 index만 업데이트 → 기존 구조를 최대한 재사용하면서 필요한 부분만 새로 생성

## 3. 고차 함수 (Higher-Order Function)

- **`map`, `filter`, `reduce`의 역할과 차이점:**
  - **`map`:** 각 요소를 다른 값으로 변환(transform) — 동일한 길이의 새로운 배열 반환 `arr.map(x => x * 2)`
  - **`filter`:** 조건에 부합하는 요소만 추출 — 일부 요소만 포함된 배열 반환 `arr.filter(x => x > 0)`
  - **`reduce`:** 모든 요소를 누적 처리하여 하나의 값으로 축약 `arr.reduce((acc, x) => acc + x, 0)`
  -
- **고차 함수를 사용했을 때의 이점:**
  - **선언적 코드:** 로직의 구현 방식(How)보다는, 무엇을 하고 싶은지(What)를 더 명확하게 표현할 수 있음
  - **코드 재사용성:** 반복 로직을 고차 함수에 추상화함으로써 중복 제거 및 재사용 가능

## 4. 시간 및 공간 복잡도 분석

- **불변 데이터 구조의 복잡도:**

  - **명령형(Mutable) 데이터 구조와의 비교:**
    | 연산 | 명령형(Mutable) | 불변(Immutable) |
    | ----------------- | ------------ | --------------------------------------- |
    | 리스트 append | O(1) | O(n) (tail node 복사 필요) |
    | 리스트 access(index) | O(n) | O(n) |
    | 해시맵 put | 평균 O(1) | O(log n) 또는 O(depth) (구조적 공유로 인해 일부 복사) |

  - **공간 복잡도:**
    - 매번 전체 구조를 복사하면 메모리 사용량이 크지만, 구조적 공유를 통해 대부분의 데이터를 재사용함으로써 효율적으로 공간을 절약함
    - 예: 100개의 노드 중 단 2개만 새로 생성 → 전체 메모리 복사가 아닌 부분 변경만 수행

## 5. 심화 학습 및 토의

### 5.1 스스로 확인할 사항

- **함수와 클로저(Closure) 비교**
    - **함수(Function)란?** 특정 작업을 수행하고, 입력에 따라 출력을 반환할 수 있는 독립적인 코드 블록입니다.
    - **클로저(Closure)란?** 자신이 생성될 때의 환경(Lexical Environment)을 기억하는 함수입니다. 즉, 외부 함수의 실행이 끝나도 해당 함수의 변수(스코프)에 접근할 수 있는 특별한 함수입니다.
    - **둘의 관계:** 클로저는 함수의 한 종류로, "상태를 기억하는 함수"라고 볼 수 있습니다. JavaScript의 모든 함수는 잠재적으로 클로저가 될 수 있지만, 외부 스코프의 변수를 참조할 때 그 특성이 발현됩니다.
    - **클로저 축약 표현:** 화살표 함수는 `function` 키워드와 `return`을 생략하여 클로저를 매우 간결하게 표현할 수 있게 해줍니다.
      ```javascript
      // 일반 함수 클로저
      function makeAdder(x) {
        return function(y) {
          return x + y;
        };
      }
      // 화살표 함수 클로저
      const makeAdderArrow = x => y => x + y;
      ```

- **내 코드의 순수 함수 분류**
    - **순수 함수 (Pure Functions):**
        - `ImmutableLinkedList`의 `item(at)`: 동일한 `at` 값에 대해 항상 동일한 데이터를 반환하며 외부 상태를 변경하지 않습니다.
        - `ImmutableHashMap`의 `_hash(key)`: 동일한 `key`에 대해 항상 동일한 해시값을 반환합니다.
        - `CampLibrary`의 `findBy(author)`, `sortByDate()`: `this.books`를 변경하지 않고, 동일한 라이브러리 상태와 인자에 대해 항상 동일한 배열을 반환합니다.
    - **순수하지 않은 함수 (Impure Functions):**
        - `CampLibrary`의 `rent(book, person)`: 내부적으로 `new Date()`를 호출하여 실행 시점마다 다른 값을 생성하므로 순수 함수가 아닙니다. (Side Effect: 시간이라는 외부 상태에 의존)
        - `ImmutableLinkedList`의 `append`, `insert`, `remove` 등: 엄밀히 말해, 이 함수들은 `new Node()`나 `new ImmutableLinkedList()`를 통해 새로운 객체를 생성하는 부작용이 있습니다. 하지만 함수형 프로그래밍의 "불변성" 관점에서는, 외부 상태를 직접 변경하지 않고 새로운 값을 반환하므로 "허용 가능한" 부작용으로 간주하며, 핵심 로직은 순수 함수처럼 동작합니다.

- **추가 고차 함수 학습**
    - **`flatMap()`:**
        - **역할:** `map()`을 수행한 후, 결과를 1단계 평탄화(flat)하는 작업을 결합한 함수입니다. 중첩 배열을 다룰 때 유용합니다.
        - **예제 코드:**
          ```javascript
          const authors = [{ name: 'Alice', books: ['Book1', 'Book2'] }, { name: 'Bob', books: ['Book3'] }];
          // map을 사용하면 [['Book1', 'Book2'], ['Book3']] 형태의 2차원 배열이 나옴
          const allBooks = authors.flatMap(author => author.books);
          // 결과: ['Book1', 'Book2', 'Book3']
          ```
    - **`every()` & `some()`:**
        - **역할:** `every()`는 배열의 **모든** 요소가 주어진 조건을 만족하면 `true`를 반환하고, `some()`은 **하나라도** 만족하면 `true`를 반환합니다.
        - **예제 코드:**
          ```javascript
          const scores = [80, 95, 72, 100];
          const allPassed = scores.every(score => score >= 70); // true
          const anyPerfect = scores.some(score => score === 100); // true
          ```
    - **`findIndex()`:**
        - **역할:** 주어진 조건을 만족하는 첫 번째 요소의 **인덱스**를 반환합니다. 만족하는 요소가 없으면 `-1`을 반환합니다. (`find`는 값 자체를 반환)
        - **예제 코드:**
          ```javascript
          const users = [{ id: 1, name: 'Alice' }, { id: 2, name: 'Bob' }];
          const bobIndex = users.findIndex(user => user.name === 'Bob'); // 1
          const charlieIndex = users.findIndex(user => user.name === 'Charlie'); // -1
          ```

### 5.2 다같이 확인할 사항

- **객체지향 vs 함수형 코드 가독성 토론**
    - **나의 기준:** (주관적 의견) 초기에는 상태와 행위가 캡슐화된 **객체지향 코드**가 이해하기 더 쉬울 수 있습니다. `library.rent(book)`처럼 현실 세계의 동작을 모방하여 직관적이기 때문입니다. 하지만 로직이 복잡해지고 여러 객체가 얽히기 시작하면, 데이터의 흐름을 추적하기 쉬운 **함수형 코드**가 더 명확하게 느껴질 수 있습니다. 데이터가 어디서 어떻게 변하는지 예측 가능하고, 파이프라인처럼 연결된 함수들의 조합으로 전체 로직을 파악하기 용이하기 때문입니다.

- **객체지향 패러다임과 함수형 패러다임 비교**
    - **공통점:** 두 패러다임 모두 복잡한 소프트웨어를 잘 구조화하여 유지보수하기 쉽고, 재사용 가능한 코드를 작성하려는 목표를 공유합니다.
    - **차이점:**
        - **데이터 처리 방식:** 객체지향은 "이 객체의 상태를 바꿔줘" (`obj.setState(...)`)라고 명령하는 반면, 함수형은 "이 데이터로 새로운 데이터를 만들어줘" (`newState = createFrom(oldState)`)라고 요청합니다. 즉, **가변성(Mutability)을 다루는 방식**이 근본적으로 다릅니다.
        - **핵심 단위:** 객체지향은 관련된 데이터와 행동을 묶은 객체(Object)가 세상의 중심입니다. 함수형은 입력과 출력이 명확한 함수(Function)가 중심이며, 데이터는 이 함수들을 통해 흐릅니다.
        - **흐름 제어:** 객체지향은 주로 메소드 호출, 상속, 다형성을 통해 프로그램의 흐름을 제어합니다. 함수형은 함수를 조합(Composition)하거나, 고차 함수에 다른 함수를 전달하는 방식으로 흐름을 만듭니다.
