# **학습 정리**

## **1. 새로 알게 된 내용**

### **- `padStart()` 와 `padEnd()` 문자열 메서드**

콘솔에 출력되는 시간표의 형식을 깔끔하게 맞추기 위해 이 메서드들을 사용했습니다. 특정 길이를 지정하고, 그 길이에 미치지 못할 경우 앞(`padStart`)이나 뒤(`padEnd`)를 특정 문자로 채워줍니다.

-   **`padStart(targetLength, padString)`**: 현재 문자열의 길이가 `targetLength`보다 짧으면, `padString`을 앞에서부터 채워 `targetLength`를 맞춥니다.
    -   **사용 예**: `String(slot.start).padStart(2, '0')` -> `9`를 `'09'`로, `10`을 `'10'`으로 만들어 시간 형식을 `HH`로 통일했습니다.
-   **`padEnd(targetLength, padString)`**: `padStart`와 유사하지만, 뒤에서부터 문자를 채웁니다.
    -   **사용 예**: `timeStr.padEnd(8)` -> `월 10-11` 같은 시간 문자열의 길이를 8자로 고정하여, 표의 전체적인 정렬을 맞췄습니다.

### **- JSDoc을 이용한 코드 문서화**

단순히 주석을 다는 것을 넘어, JSDoc이라는 정형화된 주석 문법을 사용하여 코드의 가독성과 유지보수성을 크게 향상시킬 수 있다는 것을 배웠습니다.

-   `@param {type} name - description`: 함수의 매개변수 타입, 이름, 설명을 명시합니다.
-   `@returns {type} - description`: 함수의 반환 값 타입과 설명을 명시합니다.
-   `@typedef {object} TypeName`: `TimeObject`, `Meetup`과 같이 복잡한 데이터 구조를 커스텀 타입으로 정의하여, 코드 전체에서 일관되게 참조할 수 있게 해줍니다. 이를 통해 VS Code와 같은 에디터에서 타입 힌트와 자동 완성을 지원받을 수 있었습니다.

### **- Mermaid.js를 이용한 순서도 작성**

코드의 복잡한 로직 흐름을 시각적으로 표현하기 위해 Mermaid.js 문법을 사용했습니다. 텍스트 기반으로 다이어그램을 그릴 수 있어 버전 관리가 용이하고, 마크다운 파일 내에 직접 포함시킬 수 있다는 장점을 알게 되었습니다.

## **2. 기능 구현 과정**

이번 밋업 스케줄러는 **요구사항 분석 -> 데이터 구조 설계 -> 핵심 기능 구현 -> 세부 로직 분리 -> 테스트 및 디버깅**의 체계적인 흐름으로 개발했습니다.

1.  **데이터 구조 설계**: 여러 밋업을 동시에 관리해야 한다는 요구사항에 따라, 모든 밋업 정보를 담을 전역 `meetups` 객체를 키-값 형태로 설계했습니다. 또한, 시간 문자열(`'월10-12'`)은 계산이 어렵다고 판단하여, ` { day: '월', start: 10, end: 12 }` 형태의 `TimeObject`로 파싱하여 다루는 전략을 세웠습니다. 이것이 전체 로직을 단순하게 만드는 핵심이었습니다.

2.  **핵심 기능 구현 (`arrange`, `vote`, `close`)**: 먼저 `arrange`와 `vote` 함수를 통해 데이터를 저장하는 기본 기능을 구현했습니다. 가장 복잡한 `close` 함수는 처음부터 완벽하게 만들기보다, 전체적인 처리 흐름을 먼저 구상했습니다.

3.  **`close` 함수 로직 분리 (모듈화)**: `close` 함수의 복잡도를 낮추고 '함수 10줄 제한' 요구사항을 준수하기 위해, 명확한 단일 책임을 갖는 여러 헬퍼(Helper) 함수로 기능을 분리했습니다. 데이터가 각 함수를 거치며 파이프라인처럼 처리되도록 설계했습니다.
    -   `createTimeTable`: `availableTimes`를 1시간 단위로 잘라 시간표의 뼈대를 생성합니다.
    -   `buildSchedule`: 시간표 뼈대에 `isUserAvailable` 함수로 확인한 투표 결과를 채워 넣어, 참석자 정보가 담긴 완전한 스케줄을 만듭니다.
    -   `findBestTime`: 스케줄 표를 받아 2단계의 우선순위(1. 총인원, 2. 연속 참석 인원)에 따라 최적 시간을 찾아냅니다. 특히 2순위 로직은 `findBestConsecutiveSlot`과 `calculateConsecutiveScore`라는 하위 함수로 다시 분리하여 명확성을 높였습니다.
    -   `printSchedule`: 모든 계산이 끝난 데이터를 최종 표 형식에 맞게 출력합니다.

4.  **테스트 주도 개발**: 초기 기능 구현 후, 요구사항의 예시 데이터를 바탕으로 테스트 코드를 작성하여 실행했습니다. 이후 `findBestConsecutiveSlot` 함수의 엣지 케이스를 검증하기 위해 3개의 추가 테스트 케이스를 작성하며 로직의 신뢰도를 높였습니다.

## **3. 트러블슈팅**

1.  **문제: 함수 모듈화로 인한 데이터 흐름 관리의 복잡성 증가**
    -   **현상**: '함수 10줄 제한' 규칙을 지키기 위해 `close` 함수를 여러 개의 작은 함수로 분리했습니다. 이로 인해 특정 하위 함수(`printBestTime`)에 필요한 데이터(`meetupId`)를 최상위 함수(`close`)부터 계속해서 매개변수로 전달해야 하는, 이른바 'Prop Drilling'과 유사한 상황이 발생했습니다. 초기에는 이 과정을 누락하여 하위 함수에서 `meetupId`를 하드코딩하는 실수를 저질렀습니다.
    -   **원인**: 복잡한 로직을 잘게 쪼개는 과정에서, 각 함수가 어떤 데이터를 필요로 하는지에 대한 추적을 놓쳤기 때문입니다. 모듈화는 코드의 명확성을 높이지만, 상태(데이터) 관리의 복잡성을 증가시킬 수 있다는 점을 간과했습니다.
    -   **해결**: 각 함수의 입력과 출력을 명확히 재정의하고, 데이터 흐름도를 그려보며 필요한 모든 데이터가 상위 함수에서 하위 함수로 올바르게 전달되는지 체계적으로 확인했습니다. `close` -> `printSchedule` -> `printBestTime`으로 이어지는 `meetupId`의 전달 경로를 명시적으로 수정하여 문제를 해결했습니다.

2.  **문제: 최적 시간 선정 로직의 잠재적 동점 처리 오류**
    -   **현상**: 최적 시간을 찾는 `findBestTime` 로직을 구상할 때, 요구사항에 명시된 2가지 기준(총인원, 연속 참석 인원)이 모두 동점일 경우 어떻게 처리해야 할지에 대한 ambiguity가 존재했습니다.
    -   **원인**: 요구사항에는 3순위 규칙이 명시되어 있지 않았습니다. 만약 이 경우를 고려하지 않고 코드를 작성했다면, 실행 순서에 따라 결과가 달라지는 비결정적인 코드가 되거나, 불필요하게 복잡한 3순위 규칙을 추가하게 될 수 있었습니다.
    -   **해결**: "모든 조건이 동일할 경우, 시간표 상 먼저 등장하는 시간을 우선한다"는 합리적인 내부 규칙을 세웠습니다. `findBestConsecutiveSlot` 함수에서 `>` 비교 연산자를 사용하여, 새로운 최고 점수가 등장했을 때만 `bestSlot`을 갱신하도록 구현했습니다. 이로써 점수가 동일할 때는 기존 `bestSlot`이 유지되어, 자연스럽게 먼저 등장한 시간이 최종 선택되는 결정적(deterministic) 로직을 완성했습니다.

3.  **문제: 테스트 데이터 불일치로 인한 잘못된 결과 출력**
    -   **현상**: 초기 테스트 결과, 주최자가 설정하지 않은 `수16-17` 시간대가 표에 출력되는 문제가 있었습니다.
    -   **원인**: 요구사항의 예시 데이터는 `수12-16`, `수17-18`이었으나, 테스트 코드의 `availableTimes` 배열에 `수12-18`로 잘못 입력했던 것이 원인이었습니다.
    -   **해결**: 코드를 다시 한번 꼼꼼히 검토하고, 요구사항과 테스트 데이터를 비교하여 `availableTimes` 배열을 `['월10-16', '화10-13', '수12-16', '수17-18', '금19-20']`으로 정확하게 수정하여 문제를 해결했습니다. 이를 통해 테스트 데이터의 정확성이 로직 검증에 얼마나 중요한지 깨달았습니다.
