# Day 7: 가상 파일 시스템 구현 🚀

## 1. 요구사항 분석 🧐

### 주요 기능
- **CLI (Command Line Interface)**: 사용자가 명령어를 입력하여 파일 시스템과 상호작용할 수 있는 인터페이스를 제공합니다.
- **초기화 (`init`)**: 파일 시스템의 최대 용량을 설정하고, 루트 디렉토리(`/`)만 존재하는 상태로 초기화합니다.
- **디렉토리 생성 (`mkdir`)**: 지정된 경로에 새로운 디렉토리를 생성합니다.
- **파일 생성 (`create`)**: 지정된 경로에 텍스트 파일을 생성하고, 내용을 저장합니다.
- **디렉토리 조회 (`list`)**: 특정 디렉토리 내부의 파일 및 다른 디렉토리 목록을 이름, 크기와 함께 보여줍니다.
- **파일 조회 (`read`)**: 텍스트 파일의 내용을 읽어 출력합니다.
- **영속성**: 프로그램이 종료되어도 파일 시스템의 상태(구조, 데이터)가 유지되어야 하며, 재시작 시 복원되어야 합니다.

### 추가 기능
- **가져오기/내보내기 (`import`/`export`)**: 호스트(로컬) 파일 시스템과 가상 파일 시스템 간에 파일을 복사할 수 있습니다.
- **바이너리 파일 지원**: 텍스트 파일뿐만 아니라 이미지 등 다양한 형식의 파일을 지원합니다.

## 2. 설계 📝

### 2.1. 자료구조 (In-Memory)
파일 시스템은 트리 구조로 설계합니다. 각 노드는 디렉토리 또는 파일을 나타내는 객체입니다.

- **`FileSystem` 클래스**: 전체 파일 시스템을 관리합니다. 최대 용량, 현재 사용량, 그리고 파일 시스템의 루트 노드를 포함합니다.
- **`Directory` 클래스**: 자식 노드들을 `Map` 형태로 관리합니다. (`children: Map<string, Node>`)
- **`File` 클래스**: 파일의 내용(`content`), 크기(`size`), 인코딩 방식(`encoding`)을 속성으로 가집니다.

### 2.2. 영속성 (Persistence)
- **저장**: 파일 시스템의 모든 변경(생성, 삭제, 수정)이 발생할 때마다, `FileSystem` 객체 전체를 직렬화하여 하나의 JSON 파일(`vfs.json`)에 저장합니다.
- **복원**: 프로그램 시작 시, `vfs.json` 파일이 존재하면 해당 파일을 읽어 `FileSystem` 객체를 복원합니다. 파일이 없으면 초기화 절차를 안내합니다.

## 3. 최종 명령어 구조 ⌨️

- `init <size>`: 파일 시스템을 지정된 크기(byte)로 초기화합니다.
- `mkdir <path>`: 지정된 경로에 디렉토리를 생성합니다. (예: `mkdir /documents`)
- `create <dirPath> <fileName> [content]`: 지정된 디렉토리에 텍스트 파일을 생성합니다. (예: `create /documents report.txt "회의록"`)
- `list [path]`: 해당 경로의 목록을 출력합니다. (경로 생략 시 루트)
- `read <path>`: 파일의 내용을 출력합니다.
- `import <hostPath> <vfsPath>`: 호스트의 파일을 가상 시스템으로 가져옵니다.
- `export <vfsPath> <hostPath>`: 가상 시스템의 파일을 호스트로 내보냅니다.
- `exit`: 변경사항을 저장하고 프로그램을 종료합니다.

## 4. 구현 과정 및 결과 💻

### 4.1. 미션 1: 가상 파일 시스템 CLI 구현

각 명령어의 핵심 구현 로직은 다음과 같습니다.

1.  **`init` (초기화)**
    - `new FileSystem(maxSize)`를 호출하여 메모리에 새로운 파일 시스템 객체를 생성합니다.
    - `saveFileSystem()` 함수를 호출하여 이 초기 상태를 `vfs.json` 파일에 즉시 저장합니다.

2.  **`mkdir` (디렉토리 생성)**
    - `path.dirname()`으로 부모 디렉토리 경로를, `path.basename()`으로 생성할 디렉토리 이름을 추출합니다.
    - `_findNode()` 헬퍼 함수로 부모 디렉토리 노드가 실제로 존재하는지, 그리고 그것이 디렉토리 타입인지 확인합니다.
    - 부모 디렉토리의 `children` (Map 객체)에 `new Directory(dirName)`으로 생성한 새 디렉토리 객체를 추가합니다.
    - 변경된 파일 시스템 상태를 `saveFileSystem()`으로 저장합니다.

3.  **`create` (파일 생성)**
    - `_findNode()`로 파일이 생성될 부모 디렉토리가 존재하는지 확인합니다.
    - 파일 내용의 크기를 계산하여 파일 시스템의 남은 용량이 충분한지 확인합니다.
    - `new File(fileName, content, 'utf8')`로 새 파일 객체를 생성하고 부모의 `children`에 추가합니다.
    - `usedSize`를 업데이트하고 `saveFileSystem()`으로 저장합니다.

4.  **`list` (목록 조회)**
    - `_findNode()`로 조회할 경로의 노드를 찾고, 디렉토리인지 확인합니다.
    - 디렉토리의 `children` 맵을 순회하며 각 노드의 이름, 타입, 크기를 형식에 맞춰 출력합니다.

5.  **`read` (파일 읽기)**
    - `_findNode()`로 읽을 파일 노드를 찾고, 파일 타입인지 확인합니다.
    - 파일의 `encoding` 속성을 확인하여, `base64`이면 `[Binary File]`로 출력하고, `utf8`이면 `content`를 그대로 출력합니다.

6.  **`import` & `export` (파일 입출력)**
    - **`import`**: Node.js의 `fs.readFileSync(hostPath)`로 실제 파일을 **인코딩 없이 `Buffer` 객체로** 읽습니다. 이 Buffer를 `base64` 문자열로 변환하여 `new File()` 객체를 생성하고 가상 파일 시스템에 추가합니다. 이를 통해 모든 종류의 파일을 텍스트 기반인 JSON에 안전하게 저장할 수 있습니다.
    - **`export`**: 가상 파일 시스템에서 파일 노드를 찾아 `content`와 `encoding` 속성을 가져옵니다. `Buffer.from(content, encoding)`을 통해 `content`를 다시 원본 `Buffer`로 디코딩한 후, `fs.writeFileSync(hostPath, buffer)`를 통해 실제 파일로 저장합니다.

### 4.2. 미션 2: Jest를 이용한 단위 테스트 작성

1.  **Jest 설치 및 설정**: `npm`을 통해 `jest`를 설치하고, `package.json`의 `test` 스크립트를 `jest`로 변경하여 테스트 환경을 구축했습니다.
2.  **코드 리팩토링**: 테스트의 효율성과 독립성을 높이기 위해, 기존 `vfs.js` 파일의 로직을 두 개로 분리했습니다.
    - **`vfs-core.js`**: `File`, `Directory`, `FileSystem` 클래스 등 순수한 핵심 로직만 담아 테스트 대상으로 삼기 용이하게 만들었습니다.
    - **`vfs.js`**: `readline` 기반의 CLI 로직과 `vfs-core.js`를 `require`하여 사용하는 부분만 남겼습니다.
3.  **테스트 코드 작성**: `vfs.test.js` 파일을 생성하고, `describe`, `test`, `beforeEach` 등의 Jest API를 사용하여 각 기능의 정상 케이스와 예외 케이스(예: 없는 경로에 파일 생성, 용량 초과 등)를 검증하는 테스트 코드를 작성했습니다.
4.  **테스트 실행**: `npm test` 명령어로 모든 테스트를 실행하고, 12개의 테스트 케이스가 모두 통과하는 것을 확인하여 코드의 안정성을 확보했습니다.

## 5. 💡 미션을 통해 얻은 교훈

- **명확한 요구사항의 중요성**: 초기 구현 시 표준 Unix 명령어(`touch`, `ls`)와 미션 요구사항(`create`, `list`) 간의 차이를 혼동했습니다. 명세서를 꼼꼼히 확인하고 그에 맞춰 기능을 구현하는 기본의 중요성을 다시 한번 깨달았습니다.
- **인메모리 연산과 영속성의 조화**: 모든 작업을 메모리에서 빠르게 처리하되, 변경이 발생할 때만 디스크(파일 I/O)에 기록하는 방식은 성능과 데이터 안정성을 모두 잡는 효과적인 아키텍처임을 체감했습니다. 이는 현대 애플리케이션에서 널리 사용되는 캐싱 전략과도 맞닿아 있습니다.
- **데이터 직렬화의 어려움과 해결**: `Map` 객체나 클래스 인스턴스처럼 복잡한 자료구조를 JSON으로 저장하고 복원하는 과정은 생각보다 까다로웠습니다. `replacer`와 `reviver` 함수는 이 문제를 해결하는 강력한 도구이며, 데이터의 구조를 깊이 이해해야만 올바르게 사용할 수 있다는 것을 배웠습니다.
- **테스트 주도 개발(TDD)의 가치**: 테스트 코드를 작성하기 위해 기존 코드를 리팩토링하는 과정을 거치면서, 자연스럽게 각 모듈의 역할과 책임이 명확해지는 경험을 했습니다. 이는 테스트가 단순히 버그를 잡는 것을 넘어, 더 나은 코드 설계를 유도하는 과정임을 보여주었습니다.
