# **Day 7: 학습 정리 📚**

## **1. 파일 시스템 (File System)**

### **1.1. 개념**

- 운영체제가 파일과 디렉토리를 디스크와 같은 저장 매체에 **어떻게 저장하고, 접근하고, 관리할 것인지**에 대한 규칙과 구조를 정의한 시스템입니다.
- 사용자는 파일 시스템 덕분에 물리적인 데이터 블록 주소를 몰라도, `C:\Users\...`나 `/home/...`과 같은 논리적인 경로를 통해 데이터에 접근할 수 있습니다.

### 📁 파일 시스템이 하는 일

| 역할       | 설명                                                  |
| ---------- | ----------------------------------------------------- |
| **저장**   | 데이터를 디스크에 저장할 수 있게 함                   |
| **조직화** | 파일과 디렉토리를 계층적으로 관리                     |
| **식별**   | 각 파일에 이름, 위치, 크기, 권한 등 메타데이터를 부여 |
| **접근**   | 읽기, 쓰기, 탐색 등의 동작을 가능하게 함              |

### 📚 주요 개념 정리

| 개념                     | 설명                                                          |
| ------------------------ | ------------------------------------------------------------- |
| **파일 (File)**          | 데이터의 저장 단위 (텍스트, 바이너리 등)                      |
| **디렉토리 (Directory)** | 파일 또는 다른 디렉토리를 포함하는 **폴더** 역할              |
| **루트 디렉토리 (`/`)**  | 파일 시스템의 최상위 디렉토리                                 |
| **경로 (Path)**          | 파일이나 디렉토리의 위치를 나타내는 문자열                    |
| **메타데이터**           | 이름, 크기, 생성시간, 권한, 위치 등 파일의 속성 정보          |
| **블록 (Block)**         | 파일 시스템이 데이터를 저장하는 기본 단위 (보통 512B, 4KB 등) |

### 대표적인 파일 시스템 종류

| 종류                      | 특징                                                             |
| ------------------------- | ---------------------------------------------------------------- |
| **FAT12 / FAT16 / FAT32** | MS-DOS와 Windows에서 사용됨. 구조가 단순하여 **학습용으로 적합** |
| **EXT3 / EXT4**           | Linux에서 사용하는 대표적인 파일 시스템                          |
| **NTFS**                  | Windows에서 사용하는 고급 파일 시스템                            |
| **exFAT**                 | FAT보다 향상된 범용 파일 시스템 (USB, SD카드 등에 사용)          |

### 🔎 FAT12 구조 간단히 보기

> FAT12는 디스크 공간을 클러스터 단위로 나누고, FAT 테이블을 통해 어떤 클러스터가 어떤 파일에 연결되어 있는지를 관리합니다.

### 📌 구성요소

| 영역                            | 설명                                                        |
| ------------------------------- | ----------------------------------------------------------- |
| **부트 섹터**                   | 파일 시스템 정보를 저장. 첫 번째 섹터에 있음                |
| **FAT (File Allocation Table)** | 파일이 차지하는 클러스터 체인을 저장 (연결 리스트처럼 관리) |
| **루트 디렉토리 영역**          | 루트 디렉토리의 파일/폴더 목록 저장                         |
| **데이터 영역**                 | 실제 데이터가 저장되는 공간 (파일 내용 등)                  |

### 📌 FAT 방식 저장 예시

```
파일 A :  클러스터 2 → 5 → 6 (연결된 형태로 저장됨)
파일 B :  클러스터 3 → 7
```

> FAT 테이블:
>
> ```
> 클러스터 번호:   2   3   5   6   7
> 다음 위치:       5   7   6  END  END
> ```

### 윈도우와 리눅스

| 구분          | Windows NTFS               | Linux EXT4                             |
| ------------- | -------------------------- | -------------------------------------- |
| 디렉토리 구조 | C:, D:\ 등 드라이브별 구분 | `/`부터 시작하는 단일 트리 구조        |
| 메타데이터    | 파일명, 시간, 권한 등      | inode에 모두 저장                      |
| 디스크 구조   | MFT (Master File Table)    | Superblock + inode table + block group |

## VFS (Virtual File System, 가상 파일 시스템)의 역할

> 세상에는 NTFS(Windows), APFS(Mac), ext4(Linux) 등 다양한 종류의 파일 시스템이 있습니다. 만약 Photoshop 같은 프로그램이 이 모든 파일 시스템의 작동방식을 각각 알아야 한다면 개발이 매우 복잡해질 겁니다.

이때 등장하는 것이 VFS입니다. VFS는 운영체제 커널에 있는 추상화 계층으로, 일종의 "통합 어댑터" 또는 "번역기" 역할을 합니다.

- 어떻게 동작하는가?
  1. 사용자나 프로그램이 "파일을 열어줘" (open()) 라는 표준 명령을 내립니다.
  2. VFS가 이 명령을 받습니다.
  3. VFS는 해당 파일이 저장된 디스크의 파일 시스템 종류(예: NTFS)를 확인합니다.
  4. VFS가 표준 명령을 NTFS가 알아들을 수 있는 구체적인 저수준 명령으로 변환하여 전달합니다.
- 결과적으로, 응용 프로그램은 파일 시스템의 종류와 상관없이 open(), read(), write() 같은 동일한 명령어로 모든 파일에 접근할 수 있게 됩니다.

### **FAT12 파일 시스템 구조**

- FAT12는 초창기 플로피 디스크 등에 사용되던 매우 단순한 파일 시스템입니다. 디스크를 세 개의
  구역으로 나누어 관리합니다.

1. **예약된 영역 (Reserved Region) & 부트 섹터 (Boot Sector)**

- 디스크의 가장 첫 부분입니다.
- 부트 섹터(Boot Sector)가 핵심이며, 컴퓨터가 이 디스크를 어떻게 읽어야 할지에 대한 "설명서"가 담겨 있습니다.
  - BPB (BIOS Parameter Block): 섹터당 바이트 수, 클러스터당 섹터 수, FAT 테이블의 개수 등 디스크의 물리적 사양이 기록된 중요한 정보입니다.

1. **FAT 영역 (File Allocation Table Region)**

- 이름 그대로 파일 할당 테이블(FAT)이 있는 곳입니다. 보통 데이터 손실에 대비해 2개의 동일한 복사본을 저장합니다.
- FAT는 데이터 영역을 클러스터(Cluster)라는 일정한 크기의 묶음으로 나누고, 각 클러스터의 상태를 기록하는 거대한 "지도" 또는 "장부"입니다.
- FAT12에서는 각 클러스터의 주소를 12비트로 표현합니다. (그래서 이름이 FAT12입니다.)
    <aside>
    ✅
    
    ### FAT의 동작 방식:
    
    1. 하나의 파일은 여러 개의 클러스터에 나뉘어 저장될 수 있습니다. (조각조각 흩어져 있을 수 있음)
    2. 디렉토리 엔트리(아래에서 설명)에는 파일의 첫 번째 클러스터 번호만 기록됩니다.
    3. FAT 테이블에서 그 첫 번째 클러스터 번호에 해당하는 항목을 찾아보면, 파일의 다음 조각이 담긴 클러스터 번호가 적혀 있습니다.
    4. 이런 식으로 계속 다음 클러스터를 찾아가다 보면, 파일의 마지막 클러스터에 도달하게 되고, 그곳에는 "파일 끝(End of Chain)"이라는 특별한 표시가 되어 있습니다.
    </aside>
    
    이처럼 FAT는 파일의 조각들을 연결 리스트(Linked List) 형태로 이어줍니다.


1. **데이터 영역 (Data Region)**

- 실제 파일의 내용과 디렉토리 정보가 저장되는 가장 큰 공간입니다.
- **루트 디렉토리(Root Directory):**
  - 데이터 영역의 가장 처음에 위치하며, 최상위 폴더(/ 또는 C:\)의 내용을 담습니다.
  - FAT12에서는 크기가 고정되어 있어 저장할 수 있는 파일/폴더의 개수에 제한이 있습니다.
  - 각 파일/폴더에 대한 32바이트 크기의 디렉토리 엔트리(Directory Entry) 목록으로 구성됩니다.
- **디렉토리 엔트리 (32 bytes): 파일 하나에 대한 정보 명세서**
  - 파일 이름 (8글자) + 확장자 (3글자)
  - 속성 (읽기 전용, 숨김 파일 등)
  - 생성 날짜 및 시간
  - 시작 클러스터 번호 (가장 중요!)
  - 파일 크기
- **나머지 데이터 영역:**
  - 파일의 실제 내용(데이터)과 하위 디렉토리들이 저장됩니다.
  - 하위 디렉토리 역시 그 안에 포함된 파일/폴더들의 디렉토리 엔트리 목록을 데이터로 가집니다.

## **2. 파일 I/O (Input/Output)**

### **2.1. 개념**

- 프로그램이 파일 시스템의 파일로부터 데이터를 **읽거나(Input)**, 파일에 데이터를 **쓰는(Output)** 모든 행위를 의미합니다.
- 대부분의 프로그래밍 언어는 파일 I/O를 위한 라이브러리나 내장 함수를 제공하며, 이는 내부적으로 운영체제의 시스템 콜(System Call)을 호출하여 동작합니다.

### **2.2. 저수준 vs 고수준 I/O**

- **저수준 I/O (Low-level I/O)**
  - 운영체제가 제공하는 기본적인 I/O 기능에 가깝습니다. (예: `open`, `read`, `write`, `close` 시스템 콜)
  - 데이터를 바이트(byte) 단위로 직접 다루며, 버퍼링과 같은 편의 기능을 프로그래머가 직접 구현해야 할 때가 많습니다.
  - Node.js의 `fs.open()`, `fs.read()` 등이 여기에 해당합니다.
- **고수준 I/O (High-level I/O)**
  - 저수준 I/O를 감싸서 사용하기 편리하게 만든 것입니다.
  - 버퍼링(Buffering)을 통해 I/O 성능을 최적화하고, 데이터를 줄(line) 단위나 특정 형식(JSON, XML)으로 쉽게 처리할 수 있는 API를 제공합니다.
  - Node.js의 `fs.readFile()`, `fs.createReadStream()` 등이 여기에 해당합니다.

## 3. 미션 구현을 통해 배운 핵심 기술

이번 가상 파일 시스템과 단위 테스트 미션을 통해 다음과 같은 실용적인 기술들을 학습하고 적용할 수 있었습니다.

### **3.1. Node.js 핵심 모듈 활용**

- **`fs` (File System) 모듈**: 파일 시스템의 영속성을 확보하기 위해 사용했습니다.
    - **동기 방식 활용**: `readFileSync`와 `writeFileSync`를 사용하여 프로그램을 시작할 때 `vfs.json` 파일을 동기적으로 읽어 메모리에 복원하고, 파일 시스템에 변경이 생길 때마다 즉시 파일에 저장하여 데이터의 정합성을 보장했습니다.
    - **파일 존재 확인**: `existsSync`를 통해 `vfs.json` 파일의 존재 여부를 먼저 확인하여, 파일이 없을 경우 초기화 메시지를 띄우는 분기 처리를 구현했습니다.

- **`path` 모듈**: 운영체제(Windows, macOS, Linux)에 상관없이 일관된 파일 경로를 처리하기 위해 사용했습니다.
    - **경로 결합**: `path.join()`을 사용하여 상대 경로를 절대 경로로 안전하게 변환했습니다. (`/` + `test` -> `/test`)
    - **경로 분리**: `path.dirname()`과 `path.basename()`을 사용하여 전체 경로에서 부모 디렉토리 경로와 파일/디렉토리 이름을 쉽게 분리하여 `mkdir`와 `create` 로직을 구현했습니다.

### **3.2. 데이터 직렬화 (Serialization)와 JSON**

- **개념**: 메모리 상에 존재하는 `FileSystem` 객체(클래스 인스턴스와 `Map` 자료구조)는 그 자체로 파일에 쓸 수 없습니다. 이를 파일로 저장 가능한 텍스트 형태로 변환하는 과정인 **직렬화**의 필요성을 이해했습니다.
- **`JSON.stringify`와 `replacer`**: `Map` 객체는 표준 JSON 형식으로 변환되지 않는 문제가 있었습니다. 이를 해결하기 위해 `JSON.stringify`의 두 번째 인자인 `replacer` 함수를 사용하여 `Map` 객체를 `[key, value]` 형태의 배열로 변환하여 JSON으로 저장하는 방법을 적용했습니다.
- **`JSON.parse`와 `reviver`**: 저장된 JSON 파일을 다시 읽어올 때, `replacer`로 변환했던 배열을 다시 `Map` 객체로 복원하기 위해 `JSON.parse`의 `reviver` 함수를 사용했습니다. 또한, 클래스 인스턴스 정보는 JSON에 저장되지 않으므로, 순수 객체(POJO)로 복원된 데이터를 바탕으로 `new File()`이나 `new Directory()`를 호출하여 클래스 인스턴스를 재구성하는 과정을 거쳤습니다.

### **3.3. 바이너리 데이터 처리: `Buffer`와 `Base64` 인코딩**

- **`Buffer` 객체**: Node.js가 이미지, 동영상 등 텍스트가 아닌 바이너리 데이터를 메모리에서 다루는 방식인 `Buffer`의 개념을 학습했습니다.
- **Base64 인코딩**: 바이너리 데이터는 텍스트 파일인 JSON에 직접 포함시킬 수 없습니다. 이 문제를 해결하기 위해, `import` 시 파일을 `Buffer`로 읽은 뒤 `buffer.toString('base64')`를 통해 ASCII 문자로만 구성된 안전한 텍스트로 **인코딩**하여 저장했습니다. `export` 시에는 이 Base64 문자열을 `Buffer.from(content, 'base64')`를 통해 다시 원본 바이너리 `Buffer`로 **디코딩**하여 파일로 저장함으로써 다양한 파일 형식을 지원할 수 있었습니다.

### **3.4. 단위 테스트와 Jest 프레임워크**

- **테스트의 중요성**: 기능 추가 및 코드 수정 시 발생할 수 있는 잠재적인 버그를 미리 방지하고, 코드의 안정성을 확보하기 위해 단위 테스트가 왜 필수적인지 이해했습니다.
- **테스트를 위한 리팩토링**: 테스트가 용이한 코드는 곧 잘 설계된 코드라는 것을 배웠습니다. 기존의 `vfs.js` 파일에서 순수 로직 부분(`vfs-core.js`)과 CLI 인터페이스 부분(`vfs.js`)을 분리함으로써, 외부 요인(CLI 입력 등)에 방해받지 않고 핵심 기능만 독립적으로 테스트할 수 있는 환경을 구축했습니다.
- **Jest 사용법**:
    - `describe`, `test`(또는 `it`), `beforeEach`를 사용하여 테스트 스위트와 케이스를 체계적으로 구성했습니다.
    - `expect`와 함께 `toBeInstanceOf` (클래스 타입 검증), `toBe` (정확한 값 비교), `toBeNull` (null 여부 확인) 등 다양한 Matcher 함수를 사용하여 코드의 동작을 명확하게 검증했습니다.
