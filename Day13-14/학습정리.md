### **1. Git의 3대 핵심 구성요소**

### **🔹 1-1. 객체 (Objects)**

Git은 모든 데이터를 **객체**로 저장합니다. 이 객체들의 구조를 이해해야 합니다.

- **학습 키워드**: `Blob`, `Tree`, `Commit`, `SHA-1`, `zlib`

### 📄 1) Blob 객체

- **역할**: 일반 파일의 내용을 저장
- **형식**:
    
    ```php
    blob <size>\0<content>
    ```
    
    예: `blob 13\0Hello Git World`
    
- **특징**:
    - 파일의 이름, 권한 등의 메타데이터는 저장하지 않음
    - 내용만 저장하고 중복을 제거함 (내용이 같으면 해시도 같음)

### 2) Tree 객체

- **역할**: 디렉토리의 구조를 표현하고, 하위 파일/디렉토리를 참조
- **형식**:
    
    ```php
    <mode> <filename>\0<SHA-1(binary)>
    ```
    
    - `<mode>`: 예) 100644 (일반 파일), 40000 (디렉토리)
    - `<SHA-1>`: 참조하는 blob 혹은 tree 객체의 해시 (binary)
- **예시** (ls-tree 명령으로 확인):
    
    ```bash
    100644 blob abcd1234...    hello.txt
    40000 tree 1234abcd...     src
    ```
    

### 3) Commit 객체

- **역할**: 특정 시점의 프로젝트 스냅샷을 나타냄
- **구성 요소**:
    
    ```php
    commit <size>\0
    tree <tree_sha>
    parent <parent_commit_sha>  # 최초 커밋이 아닌 경우
    author <name> <email> <timestamp> <timezone>
    committer <name> <email> <timestamp> <timezone>
    
    <commit message>
    ```
    
- **특징**:
    - `tree`는 현재 커밋의 전체 프로젝트 디렉토리 구조를 가리킴
    - `parent`는 바로 이전 커밋 (이중 커밋 시 머지 커밋은 2개 이상)
    - `author`와 `committer`는 동일한 경우가 대부분

### Git 객체의 저장 방식

- 모든 객체는 `blob`, `tree`, `commit` 타입에 관계없이 다음과 같은 순서로 저장됩니다:

### 저장 과정:

1. 객체 원본 생성 (예: `blob 12\0Hello World\n`)
2. SHA-1 해시 생성 → 해시가 객체의 **ID**가 됨
3. zlib 압축
4. `.git/objects/xx/yyyyyy…` 경로에 저장
    - `xx`: 해시의 앞 2자리
    - `yyyyyy…`: 해시의 나머지 38자리

### 🔧 실습 예제: `.git/objects` 내부 파일 읽기

아래는 특정 Git 객체 파일을 읽고 압축을 해제하고, 원래 내용을 출력하는 Node.js 코드입니다.

```php
const fs = require('fs');
const zlib = require('zlib');
const path = require('path');

// Git 객체 해시의 앞 2자리와 나머지를 나눔
function readGitObject(hash) {
  const gitDir = path.join(process.cwd(), '.git');
  const objectPath = path.join(gitDir, 'objects', hash.slice(0, 2), hash.slice(2));

  const compressed = fs.readFileSync(objectPath);
  const decompressed = zlib.inflateSync(compressed);

  const content = decompressed.toString('utf-8');
  const [header, body] = content.split('\0');
  const [type, size] = header.split(' ');

  console.log(`🔹 Type: ${type}`);
  console.log(`🔹 Size: ${size}`);
  console.log(`📦 Content:\n${body}`);
}


readGitObject('e69de29bb2d1d6434b8b29ae775ad8c2e48c5391'); 

```

```php
🔹 Type: commit
🔹 Size: 224
📦 Content:
tree 4ea50c2f396b56fa433eb007ea4484b9e0e28622
parent 6d9f9aac91e3ec9acf1e611a9c3e483a773e3ebc
author Ss0Mae <ssomae@protonmail.ch> 1753857892 +0900
committer Ss0Mae <ssomae@protonmail.ch> 1753857892 +0900

feat: gitlog test
```

### **🔹 1-2. 인덱스 (Index): 스테이징 영역의 실체**

Git에서 `git add` 명령을 실행하면 파일이 인덱스(index)에 기록되고, 그 이후 `git commit`을 통해 커밋 객체가 만들어집니다. 즉, 인덱스는 커밋 직전의 준비 단계로, **다음 커밋이 어떤 파일을 포함할지 정리된 목록**입니다

- **학습 키워드**: `Staging Area`, `Cache`, `.git/index`, `Binary File Parsing`

### ✅ 인덱스 파일의 구조

인덱스 파일은 `.git/index` 경로에 존재하며, **바이너리 형식의 파일**입니다. 이 파일은 크게 3개의 주요 파트로 구성되어 있습니다.

### 전체 구조

```php
[Header]
[Entry 1]
[Entry 2]
...
[Checksum]
```

### 📌 1) Header (12 bytes)

| Offset | 크기(bytes) | 의미 |
| --- | --- | --- |
| 0 | 4 | 시그니처 `DIRC` |
| 4 | 4 | 버전 (2 또는 3) |
| 8 | 4 | 엔트리 개수 |

### 📌 2) Entry (각 항목마다 62 bytes + 가변)

각 엔트리는 하나의 파일을 나타내며, 다음과 같은 필드들로 구성됩니다

- Entry는 반드시 8바이트로 정렬되며, 패딩을 포함합니다.

| Offset | 크기(bytes) | 필드명 |
| --- | --- | --- |
| 0 | 4 | ctime (생성 시간, UNIX timestamp) |
| 8 | 4 | mtime (수정 시간) |
| 16 | 4 | dev (디바이스) |
| 20 | 4 | ino (inode 번호) |
| 24 | 4 | mode (권한) |
| 28 | 4 | uid |
| 32 | 4 | gid |
| 36 | 4 | file size |
| 40 | 20 | SHA-1 (해당 파일의 blob 객체 ID) |
| 60 | 2 | flags |
| 62 | ? | 경로명 (null-terminated, 8의 배수로 패딩) |

### 인덱스 파일의 헤더 읽기 (Node.js)

```php
const fs = require('fs');

function readIndexHeader() {
  const data = fs.readFileSync('.git/index');

  const signature = data.toString('utf-8', 0, 4);
  const version = data.readUInt32BE(4);
  const entryCount = data.readUInt32BE(8);

  console.log(`🔹 Signature: ${signature}`); // DIRC
  console.log(`🔹 Version: ${version}`);     // usually 2
  console.log(`📦 Entry Count: ${entryCount}`);
}

readIndexHeader();

```

```php
🔹 Signature: DIRC
🔹 Version: 2
📦 Entry Count: 6
```

### 인덱스의 핵심 역할

- `Working Directory`의 변경사항을 추적하고 `git add`를 통해 커밋할 준비를 함
- `git diff`는 `index`와 `working directory`, 또는 `index`와 `HEAD`를 비교함
- `git checkout`, `git reset`, `git stash` 등 대부분의 명령이 index를 참조함

### 요약

| 요소 | 설명 |
| --- | --- |
| `.git/index` | 다음 커밋에 포함될 파일들의 정보가 담긴 바이너리 파일 |
| Header | 시그니처(`DIRC`), 버전, 엔트리 수를 포함 |
| Entry | 각 파일의 SHA-1, 파일명, 권한, 시간 등의 메타정보 포함 |

### **🔹 1-3. 참조 (References)**

`main` 브랜치, `HEAD` 포인터 등은 모두 **참조**라는 간단한 파일로 관리됩니다.

- **학습 키워드**: `Refs`, `HEAD`, `Branches`, `Tags`

### ✅ Git 참조 시스템의 구성

### 📁 주요 파일 위치

| 참조 종류 | 경로 예시 | 설명 |
| --- | --- | --- |
| 브랜치 | `.git/refs/heads/main` | 현재 브랜치가 가리키는 커밋 해시 |
| 태그 | `.git/refs/tags/v1.0` | 태그가 가리키는 커밋 해시 |
| HEAD | `.git/HEAD` | 현재 작업 중인 브랜치 or 직접 커밋 해시 |
| packed-refs | `.git/packed-refs` | 참조가 많을 경우 최적화를 위해 여기 저장됨 |

### 📌 1) HEAD란 무엇일까

`HEAD`는 **현재 작업 중인 브랜치**를 가리키는 *포인터*입니다.

### 🔸 HEAD 예시

- **브랜치를 가리키는 경우** (일반적인 상태):
    
    ```bash
    ref: refs/heads/main
    ```
    
- **Detached HEAD 상태** (직접 커밋을 가리키는 경우):
    
    ```scss
    2d3acf90b... (커밋 해시)
    ```
    

> 👉 커밋 해시를 직접 HEAD에 적으면 Detached HEAD 상태라고 부르고, 브랜치와 연결되지 않은 임시 작업 상태입니다.
> 

### 📌 2) 브랜치 파일 (refs/heads/*)

- `.git/refs/heads/main` 파일에는 단순히 **커밋의 SHA-1 해시값**이 저장되어 있습니다.
    
    ```
    4b825dc642cb6eb9a060e54bf8d69288fbee4904
    ```
    
- 이 값이 해당 브랜치의 **최신 커밋**을 나타냅니다.
- 새로운 커밋이 생성되면 Git은 이 파일을 **새로운 커밋의 해시로 업데이트**합니다.

---

### 📌 3) 커밋 순회: `git log`의 원리

- `HEAD → refs/heads/main → <커밋 SHA>` 로 따라가서 최신 커밋을 찾고,
- 커밋 객체 내부의 `parent <이전 커밋 SHA>` 항목을 따라가며 과거 커밋을 순회합니다.

> 🎯 Git은 포인터 체인을 통해 버전 히스토리를 탐색합니다.
> 

### 실습 예제: HEAD와 브랜치 해시 읽기

```jsx

const fs = require('fs');
const path = require('path');

function readHeadCommit() {
  const gitDir = path.join(process.cwd(), '.git');
  const headPath = path.join(gitDir, 'HEAD');

  const headContent = fs.readFileSync(headPath, 'utf-8').trim();

  if (headContent.startsWith('ref:')) {
    const refPath = headContent.split(' ')[1];
    const fullRefPath = path.join(gitDir, refPath);
    const commitHash = fs.readFileSync(fullRefPath, 'utf-8').trim();

    console.log(`🧭 현재 브랜치: ${refPath.split('/').pop()}`);
    console.log(`🔗 최신 커밋 해시: ${commitHash}`);
  } else {
    console.log(`🧷 Detached HEAD 상태입니다.`);
    console.log(`🔗 현재 HEAD 커밋 해시: ${headContent}`);
  }
}

readHeadCommit();
```

```jsx
🧭 현재 브랜치: main
🔗 최신 커밋 해시: 7131099776e933fda41c44005ec27a2a93b545fb
```

### 요약

| 요소 | 설명 |
| --- | --- |
| `HEAD` | 현재 브랜치를 참조하거나 detached 상태에서 직접 커밋을 가리킴 |
| `refs/heads/*` | 브랜치별로 최신 커밋 SHA-1을 담은 참조 파일 |
| `refs/tags/*` | 특정 커밋에 이름표를 붙이는 태그 참조 |
| `packed-refs` | 많은 참조를 효율적으로 관리하기 위한 압축 파일 |

### **2. 필요한 Node.js 기술**

- **`fs` (File System)**: `.git` 디렉토리의 파일을 읽기 위한 필수 모듈. 특히 바이너리 파일을 다루기 위해 `fs.readFileSync`의 동작을 이해해야 합니다.
- **`zlib`**: Git 객체의 압축을 풀기 위해 사용합니다. `zlib.inflateSync()` 또는 `zlib.unzipSync()` 함수를 사용하게 됩니다.
- **`crypto`**: 파일 내용으로 SHA-1 해시를 직접 생성해볼 때 필요합니다. `crypto.createHash('sha1')`를 사용합니다.
- **`process.argv`**: 사용자가 `node my-git.js add file.txt` 와 같이 입력한 커맨드라인 인자를 받아오기 위해 필요합니다.

### **2. `git clone`의 원리 탐구: 로컬 복제를 통해 배우기**

`git clone`은 원격 저장소의 모든 것을 로컬로 가져오는 강력한 명령어입니다. 전체 네트워크 프로토콜을 구현하기 전에, **로컬에 있는 저장소를 다른 디렉토리로 복제**하는 과정을 통해 그 핵심 원리를 학습합니다.

- **학습 키워드**: `Git Protocol`, `Smart/Dumb Protocol`, `Packfile`, `Checkout`

#### **로컬 `clone`의 핵심 단계**

1.  **저장소 초기화**: 새로운 디렉토리에 `.git` 폴더를 생성합니다 (`init`).
2.  **객체 복사**: 원본 저장소의 `.git/objects` 디렉토리 내 모든 객체들을 새로운 저장소로 복사합니다. 이것이 Git의 모든 데이터(모든 버전의 파일, 디렉토리 구조, 커밋 히스토리)를 가져오는 과정입니다.
3.  **참조(브랜치) 복사**: 원본의 `.git/refs/heads`에 있는 모든 브랜치 파일을 새로운 저장소에 그대로 복사하여 모든 브랜치를 재현합니다.
4.  **`HEAD` 설정**: 원본의 `HEAD`가 가리키던 브랜치(예: `main`)를 찾아, 새로운 저장소의 `HEAD`도 동일한 브랜치를 가리키도록 설정합니다. (`ref: refs/heads/main`)
5.  **워킹 디렉토리 생성 (Checkout)**: 마지막으로, `HEAD`가 가리키는 커밋의 `tree` 객체를 기준으로 실제 파일들을 워킹 디렉토리에 펼쳐놓습니다. 이 과정이 없다면 사용자는 빈 폴더만 보게 될 것입니다.

#### **🤔 Smart Protocol vs. Dumb Protocol**

실제 `git clone`은 HTTP나 SSH를 통해 원격 서버와 통신합니다. 이때 두 가지 방식이 사용될 수 있습니다.

- **Dumb Protocol (단순 프로토콜)**: 서버는 단순히 파일들을 제공만 하고, 클라이언트가 어떤 객체가 필요한지 모두 파악해서 요청해야 합니다. 비효율적이지만 간단한 웹서버로도 구현이 가능합니다.
- **Smart Protocol (지능형 프로토콜)**: 서버에 Git을 알고 있는 프로세스가 실행되어, 클라이언트와 효율적으로 통신합니다. 클라이언트는 자신이 가진 객체 목록을 알려주고, 서버는 클라이언트에게 없는 객체만 모아서 팩파일(Packfile)이라는 압축된 파일 형태로 전달해줍니다. 이는 네트워크 부하를 최소화하는 매우 효율적인 방식이며, 현재 대부분의 Git 작업이 이 프로토콜을 사용합니다.

이번 미션의 로컬 복제는 Dumb Protocol과 유사하게 단순히 파일들을 복사하는 방식이지만, `clone`의 근본적인 목표인 '저장소의 모든 데이터를 이전하고, 특정 버전을 기준으로 워킹 디렉토리를 구성한다'는 핵심 개념을 이해하는 데 큰 도움이 될 것입니다.
