### Day 16-17: 미션 2 구현 내용 정리 (피어세션 공유용)

**작성자: J251**

### 1. 개요

이번 미션의 목표는 `test.csv`의 대용량 질문 데이터를 벡터로 변환하여, 의미적으로 유사한 질문을 찾는 시스템을 구축하는 것이었습니다. 구현 과정에서 50만 개가 넘는 질문 데이터로 인해 발생하는 **메모리 관리 문제**가 핵심 과제였으며, 이를 해결하기 위해 **스트림(Stream) 기반 아키텍처**를 도입하여 시스템의 안정성과 확장성을 확보했습니다.

### 2. 벡터화 방식에 대한 설계 결정: 왜 고차원 임베딩 모델을 선택했는가?

미션 요구사항에서는 "관련도 - 노출빈도 - 거리"를 기반으로 벡터를 생성하는 방식을 제안했습니다. 하지만 저희는 더 현대적이고 효과적인 접근 방식을 채택했습니다.

#### 가. 전통적 방식 vs 현대적 방식

| 구분 | 전통적 방식 (특징 공학) | **채택한 방식 (사전 훈련 모델)** |
| :--- | :--- | :--- |
| **핵심 아이디어** | 개발자가 직접 단어 중복률, 출현 빈도 등<br>중요하다고 생각하는 특징(Feature)을 계산하여 벡터 생성 | 거대 언어 모델이 수많은 텍스트를 통해<br>이미 학습한 **문맥적 의미**를 벡터로 추출 |
| **벡터의 의미** | 계산된 특징 값들의 조합<br>(예: `[중복단어수, TF-IDF값, ...]`) | 문장 전체의 의미를 압축한 **384차원의 의미 좌표** |
| **장점** | 구현이 직관적이고, 벡터의 각 차원이 무엇을 의미하는지<br>해석하기 용이함 | **문맥, 뉘앙스, 동의어**까지 파악하여<br>의미적으로 훨씬 정교한 유사도 측정이 가능 |
| **단점** | "노트북 추천"과 "랩탑 추천"처럼 표현이 다르면<br>유사도를 놓치기 쉬움. 신조어나 복잡한 문맥 파악 불가. | 벡터의 각 차원이 구체적으로 어떤 의미인지<br>직접 해석하기는 어려움 (추상화 수준이 높음) |

#### 나. 최종 결정의 이유

결론적으로, "의미있는 혹은 유용한 벡터값을 제대로 저장하는 것에 집중"하라는 미션의 핵심 목표를 달성하기 위해, **사전 훈련된 `Xenova/all-MiniLM-L6-v2` 모델**을 선택했습니다. 이 모델은 단순히 단어의 출현 여부를 넘어, **문장 전체의 의미를 깊이 이해하고 벡터로 변환**해주기 때문에, 훨씬 더 높은 품질의 유사도 검색 결과를 얻을 수 있습니다. 이는 현재 자연어 처리 분야에서 의미 기반 검색을 구현하는 가장 표준적이고 효과적인 방식입니다.

### 3. 시스템 전체 로직 및 아키텍처 상세 설명

대용량 데이터 처리 과정에서 발생한 **메모리 초과(Heap out of memory) 오류**를 해결하는 것이 이 시스템 아키텍처의 핵심입니다. 전체 데이터 파이프라인은 4단계로 구성되며, 각 단계는 메모리 효율성을 극대화하기 위해 스트림(Stream) 방식으로 설계되었습니다.

---

#### **1단계: 데이터 생성 (`embed-use.js`) - "생성과 동시에 디스크에 기록"**

-   **핵심 역할**: 원본 CSV에서 질문을 추출하여, 각 질문을 고차원 벡터로 변환하고, 이를 스트림 방식으로 `use-embeddings.ndjson` 파일에 저장합니다.

-   **상세 로직**:
    1.  **모델 로딩**: `@xenova/transformers` 라이브러리의 `pipeline` 함수를 호출하여 `Xenova/all-MiniLM-L6-v2` 모델을 비동기적으로 로드합니다. 이 모델은 문장을 입력받아 384차원의 벡터(임베딩)를 출력하는 역할을 합니다.
    2.  **CSV 스트림 파싱**: `fs.createReadStream`으로 `test.csv` 파일에 대한 읽기 스트림을 생성합니다. 이 스트림을 `csv-parse` 라이브러리에 파이핑하여, 대용량 CSV 파일을 한 번에 메모리에 올리지 않고 한 줄씩 효율적으로 읽어옵니다.
    3.  **고유 질문 추출**: `Set` 자료구조를 사용하여 `question1`과 `question2` 컬럼의 모든 질문을 저장합니다. `Set`은 중복을 허용하지 않으므로, 자연스럽게 전체 데이터셋에서 고유한 질문 목록(약 52만 개)을 만들어냅니다.
    4.  **배치 처리 루프**: 52만 개의 질문을 한 번에 모델에 넣으면 메모리 문제가 발생할 수 있으므로, `batchSize`(예: 4096개) 단위로 잘라서 처리하는 루프를 실행합니다.
    5.  **임베딩 생성**: 각 배치(질문 4096개 묶음)를 모델(`feature()`)에 전달합니다. `pooling: 'mean'`과 `normalize: true` 옵션을 통해 문장 전체의 평균 의미를 담는 고품질 단위 벡터를 생성합니다.
    6.  **스트림 방식 파일 쓰기**: 생성된 임베딩 결과를 다시 메모리의 거대한 배열에 담는 것이 아니라, **`for` 루프 안에서** 각 질문과 그에 해당하는 벡터를 `{question, embedding}` 형태의 JSON 문자열로 만듭니다. 그리고 `fs.appendFileSync`를 사용해 이 문자열을 `use-embeddings.ndjson` 파일의 **끝에 즉시 추가**합니다. 이 과정을 통해 메모리 사용량을 최소화하며 결과를 파일에 기록합니다.

---

#### **2단계: 대용량 데이터 변환 (`load-embeddings.js`) - "메모리를 거치지 않는 디스크 간 이동"**

-   **핵심 역할**: `ndjson` 형식의 임베딩 파일을, 다른 시스템에서 활용하기 용이한 표준 JSON 배열 형식의 `vectors.json` 파일로 **메모리 문제 없이** 변환합니다.

-   **상세 로직**:
    1.  **문제 인식**: `ndjson` 파일의 모든 줄을 읽어 메모리 상의 배열에 `push`한 뒤, 마지막에 `JSON.stringify`를 호출하는 방식은 2GB가 넘는 데이터를 처리할 때 힙 메모리 초과를 유발합니다.
    2.  **스트림 파이프라인 구축**:
        -   `fs.createReadStream(inputPath)`: `use-embeddings.ndjson` 파일을 읽는 스트림을 생성합니다.
        -   `readline.createInterface`: 읽기 스트림을 받아, 한 줄씩 데이터를 읽어오는 인터페이스를 설정합니다.
        -   `fs.createWriteStream(outputPath)`: 최종 결과물인 `vectors.json`에 데이터를 쓸 스트림을 생성합니다.
    3.  **수동 JSON 배열 생성**:
        -   쓰기 스트림에 먼저 여는 대괄호 `[`와 줄바꿈 문자 `\n`를 씁니다.
        -   `for await...of` 구문으로 `readline`이 읽어오는 **각 줄(line)에 대해 루프를 실행**합니다.
        -   루프 안에서, 읽어온 한 줄의 텍스트를 `JSON.parse`로 객체화하고, 이를 `[질문, 벡터]` 형태의 배열로 다시 `JSON.stringify` 합니다.
        -   `isFirstLine` 플래그를 사용하여, 첫 번째 줄이 아닐 경우에만 객체 앞에 쉼표 `,`와 줄바꿈 문자를 먼저 써주어 유효한 JSON 배열 형식을 유지합니다.
        -   변환된 문자열을 쓰기 스트림에 씁니다.
    4.  **마무리**: 루프가 모두 끝나면, 닫는 대괄호 `]`와 줄바꿈 문자를 파일에 쓰고 `outputStream.end()`를 호출하여 스트림을 종료합니다. 이 전체 과정은 데이터를 메모리에 쌓아두지 않고 디스크에서 디스크로 직접 전달하는 효과를 가집니다.

---

#### **3단계: 데이터 로딩 및 DB 구축 (`main.js`) - "거대한 JSON을 조각내어 DB에 삽입"**

-   **핵심 역할**: 2GB가 넘는 `vectors.json`을 스트림으로 안전하게 읽어와 `VectorDB`를 구축합니다.

-   **상세 로직**:
    1.  **문제 인식**: `fs.readFileSync`로 `vectors.json`을 읽으면 파일 크기 제한 및 메모리 문제로 즉시 오류가 발생합니다.
    2.  **해결책: `stream-json` 라이브러리**: 이 라이브러리는 거대한 JSON 파일을 스트림으로 처리하여, 전체를 파싱하지 않고도 내부 요소에 접근할 수 있게 해줍니다.
    3.  **스트림 체이닝**:
        -   `fs.createReadStream`: 파일 시스템에서 `vectors.json`을 바이트 스트림으로 읽기 시작합니다.
        -   `.pipe(parser())`: 읽어온 바이트 스트림을 `stream-json`의 `parser`에 연결합니다. `parser`는 바이트를 분석하여 JSON의 기본 단위(토큰)로 변환합니다.
        -   `.pipe(streamArray())`: 파싱된 토큰 스트림을 `streamArray`에 연결합니다. `streamArray`는 JSON 배열(`[]`) 구조를 인식하고, 배열 안의 각 최상위 요소(우리의 경우 `[질문, 벡터]`)가 완성될 때마다 `data` 이벤트를 발생시킵니다.
    4.  **점진적 DB 구축**:
        -   `stream.on('data', ...)`: `streamArray`가 배열의 요소 하나를 완전히 읽을 때마다 이 이벤트 리스너가 호출됩니다.
        -   리스너 내부에서는 전달받은 데이터(`{key, value}`)에서 `value` (즉, `[질문, 벡터]` 배열)를 추출합니다.
        -   `db.vectors.set(sentence, new Float32Array(embedding))`: 추출한 데이터를 즉시 `Word2VecVectorDB`의 `vectors` 맵(Map)에 삽입합니다.
        -   이 과정이 52만 번 반복되며, 모든 데이터가 메모리에 한 번에 올라오지 않고 **하나씩 순차적으로 DB에 안전하게 안착**합니다.

---

#### **4단계: 실시간 유사도 검색 (`Word2VecVectorDB.js`) - "즉석에서 생성된 벡터로 전체 DB 탐색"**

-   **핵심 역할**: 이미 메모리에 구축된 DB를 기반으로, 사용자가 새롭게 입력한 질문에 대한 유사도 검색을 실시간으로 수행합니다.

-   **상세 로직**:
    1.  **실시간 검색어 임베딩**:
        -   사용자가 검색어(예: "How can I learn guitar?")를 입력하면, `Word2VecVectorDB.js`의 `getEmbeddings` 메서드가 호출됩니다.
        -   이 메서드는 이미 로드되어 메모리에 상주 중인 `transformer.js` 모델을 이용해, 이 새로운 검색어 문장 **단 하나만을 위한** 384차원 벡터를 즉시 생성하여 반환합니다. 이 과정은 매우 빠릅니다.
    2.  **전체 스캔 기반 유사도 검색 (`search`)**:
        -   `search` 메서드는 방금 생성된 '검색어 벡터'를 인자로 받습니다.
        -   DB의 `vectors` 맵(Map)에 저장된 52만 개의 모든 벡터를 `for...of` 루프를 통해 순회합니다. (현재 구현은 가장 간단한 방식인 **Full Scan**입니다.)
        -   루프의 각 단계에서, 현재 순회 중인 벡터와 '검색어 벡터' 간의 **코사인 유사도**를 `cosineSimilarity` 함수를 통해 계산합니다. 코사인 유사도는 두 벡터가 이루는 각도를 기반으로 방향성의 유사성을 측정하며, -1에서 1 사이의 값을 가집니다.
        -   계산된 모든 유사도 점수를 `{ sentence, similarity }` 형태의 객체로 만들어 새로운 배열에 저장합니다.
    3.  **결과 정렬 및 반환**:
        -   유사도 점수를 담은 배열을 `similarity` 값을 기준으로 내림차순 정렬합니다.
        -   `slice(0, 5)`를 통해 가장 점수가 높은, 즉 가장 유사한 상위 5개의 질문만 잘라내어 최종 결과로 반환합니다.

### 4. 결론

초기 구현의 메모리 문제를 해결하기 위해, 데이터 생성부터 로딩까지 전 과정을 **스트림 기반 아키텍처**로 재설계했습니다. 이를 통해 대용량 데이터도 안정적으로 처리할 수 있는 확장성을 확보했으며, 최신 언어 모델을 활용하여 높은 품질의 의미 기반 검색 기능을 성공적으로 구현할 수 있었습니다.
