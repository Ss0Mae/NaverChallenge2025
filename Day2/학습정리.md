# Day 2: 학습 정리

## 1. 가상 환경(Virtual Environment)의 중요성

VMware와 같은 가상화 도구를 사용하여 로컬 컴퓨터에 독립된 Ubuntu Server 환경을 구축했습니다. 이를 통해 실제 서버와 유사한 환경에서 마음껏 실습하며, 시스템 설정 변경이나 패키지 설치가 로컬 PC에 영향을 주지 않는 안전한 실험 공간을 확보할 수 있었습니다. OS 클린 설치부터 네트워크, 사용자 권한 등 서버의 가장 기초적인 부분을 직접 다루는 경험을 할 수 있었습니다.

## 2. 쉘 스크립트와 Crontab을 이용한 자동화

- **쉘 스크립트 (Shell Script):**
  - 반복적인 리눅스 명령어들을 하나의 파일(`.sh`)로 묶어 순차적으로 실행하는, 서버 관리 자동화의 핵심 기술입니다.
  - `#!/bin/bash`는 스크립트를 bash 쉘로 실행하라고 OS에 알려주는 약속(Shebang)입니다.
  - `chmod +x [스크립트 파일]` 명령으로 실행 권한을 부여해야만 스크립트로써 실행될 수 있습니다.

- **Crontab (Cron Table):**
  - 특정 시간에 특정 명령어나 스크립트를 주기적으로 실행하도록 예약하는 리눅스의 강력한 스케줄러입니다.
  - `crontab -e`로 예약 작업을 편집하고, `crontab -l`로 현재 등록된 작업을 확인합니다.
  - `* * * * * [명령]` 형식으로 실행 주기를 설정하며, 각 `*`는 분, 시, 일, 월, 요일을 의미합니다. 이번 미션에서는 1분마다, 그리고 매시간 정각에 작업을 실행하도록 설정했습니다.

## 3. 상세 구현 과정 및 학습 내용

이번 미션은 두 개의 핵심 스크립트를 작성하고 `cron`으로 자동화하는 과정이었습니다. 각 단계에서 사용된 명령어와 그 의미를 통해 많은 것을 배울 수 있었습니다.

### **1단계: 주기적인 로그 수집 (`collect_logs.sh`)**

단순히 로그를 수집하는 것을 넘어, 안정적으로 동작하는 스크립트를 만드는 것이 목표였습니다.

-   **타임스탬프 생성:** `TIMESTAMP=$(date +"%Y-%m-%d_%H-%M-%S")`
    -   `date` 명령어에 `+` 포맷을 지정하여, 로그 파일명이 서로 겹치지 않고 시간 순으로 정렬되도록 고유한 파일명을 생성했습니다.
-   **로그 수집 및 리디렉션:** `sudo dmesg > "$LOG_DIR/$TIMESTAMP.log"`
    -   `dmesg`는 커널 로그를 확인하는 명령어로, 일반 사용자에게는 실행 권한이 없어 `sudo`를 사용해야 했습니다.
    -   `>` 연산자는 명령어의 표준 출력(stdout)을 파일로 보내는 **리디렉션(Redirection)** 입니다. 이를 통해 `dmesg`의 결과를 화면이 아닌 파일에 저장할 수 있었습니다.
-   **에러 로깅:** `... 2>> "$LOG_DIR/cron_error.log"`
    -   `2>>`는 표준 에러(stderr)를 파일에 **덧붙이는(append)** 리디렉션입니다. `cron` 환경에서는 오류가 발생해도 화면에 보이지 않기 때문에, 에러 로그를 별도 파일로 남기는 것은 디버깅에 필수적이었습니다. 이 덕분에 `sudo` 권한 문제나 오타를 빠르게 찾을 수 있었습니다.
-   **자동화:** `crontab -e`를 통해 `* * * * * /home/test/collect_logs.sh` 등록
    -   매 분마다 스크립트가 실행되도록 설정하여, 시스템 로그를 꾸준히 축적하는 기반을 마련했습니다.

### **2단계: 로그 분석, AI 연동 및 알림 (`analyze_and_notify.sh`)**

수집된 데이터를 가공하여 의미 있는 정보로 만들고, 외부 서비스와 연동하는 복합적인 과정이었습니다.

-   **최근 로그 파일 취합:** `find ... -mmin -60 -exec cat {} +`
    -   `find` 명령어는 특정 조건의 파일을 찾는 강력한 도구입니다. `-mmin -60` 옵션으로 '최근 60분 내에 수정된 파일'만 정확히 찾아냈고, `-exec cat {} +`를 통해 찾아낸 모든 파일의 내용을 하나로 합쳤습니다.
-   **핵심 이벤트 추출 (파이프라인의 힘):** `sort | uniq -c | sort -nr | head -n 1`
    -   리눅스 **파이프라인(`|`)** 의 강력함을 체감한 부분이었습니다. 여러 명령어의 출력을 다음 명령어의 입력으로 연결하여 복잡한 데이터 처리를 한 줄로 해결했습니다.
        1.  `sort`: 모든 로그 라인을 알파벳 순으로 정렬하여 동일한 라인을 붙여줍니다.
        2.  `uniq -c`: 연속된 중복 라인을 하나로 합치고, 앞에 중복 횟수(`-c`)를 붙여줍니다.
        3.  `sort -nr`: 중복 횟수를 기준으로 숫자 내림차순(`-nr`) 정렬하여 가장 빈번한 로그를 맨 위로 올립니다.
        4.  `head -n 1`: 맨 위의 한 줄만 선택하여 최종적으로 가장 빈번한 이벤트를 추출합니다.
-   **AI 연동 (API 호출):** `curl -s ... -d "$OLLAMA_PAYLOAD"`
    -   `curl`은 터미널에서 HTTP 요청을 보내는 도구입니다. `-X POST`로 요청 방식을 지정하고, `-d` 옵션으로 보낼 데이터(Payload)를 전달했습니다.
    -   **`jq`의 활용:** `jq`는 JSON 데이터를 다루는 필수 도구였습니다. `jq -nc ...`를 사용해 쉘 변수를 JSON 형식의 문자열로 안전하게 변환하여 `OLLAMA_PAYLOAD`를 구성했습니다. API 응답에서도 `jq -r .response`를 통해 필요한 `response` 필드의 값만 깔끔하게 추출할 수 있었습니다.
-   **프로세스 자원 모니터링:** `pgrep -f "ollama serve"` 및 `ps -p $PID ...`
    -   `pgrep -f`로 특정 패턴을 포함하는 프로세스의 ID(PID)를 찾고, `ps -p $PID`에 `-o %cpu,%mem` 옵션을 주어 해당 프로세스의 CPU와 메모리 사용량만 정확히 조회했습니다. 이를 통해 특정 서비스의 상태를 모니터링하는 방법을 익혔습니다.
-   **Slack 알림 전송:**
    -   Ollama 연동과 마찬가지로 `curl`과 `jq`를 사용하여, 여러 줄로 구성된 분석 결과를 Slack 웹훅 URL로 전송했습니다. 쉘 스크립트 내에서 ``을 이용해 줄바꿈이 포함된 문자열을 만드는 방법을 활용했습니다.


## 4. 주요 트러블슈팅

- **문제 1: `cron`에서 `dmesg` 권한 오류**
    - **현상:** `cron`으로 실행된 스크립트가 로그를 수집하지 못하고 0바이트 파일만 생성했습니다.
    - **원인:** 일반 사용자 계정은 커널 로그를 읽는 `dmesg` 실행 권한이 없으며, `cron`의 비대화형(non-interactive) 환경에서는 `sudo`의 비밀번호를 입력할 수 없어 권한 상승에 실패했습니다.
    - **해결:** `sudo visudo` 명령으로 `sudoers` 파일을 열어, `test ALL=(ALL) NOPASSWD: /usr/bin/dmesg` 규칙을 추가했습니다. 이를 통해 `test` 계정이 `dmesg` 명령어에 한해서만 비밀번호 없이 `sudo`를 사용하도록 허용하여 문제를 해결했습니다.

- **문제 2: 스크립트 오류 및 디버깅의 어려움**
    - **현상:** 스크립트에 오타(`dmseg`)가 있거나 필요한 패키지(`jq`)가 없는 등, 터미널에서는 잘 되던 스크립트가 `cron`에서는 동작하지 않았습니다.
    - **원인:** `cron`은 터미널과 실행 환경이 달라 오류 메시지를 직접 볼 수 없어 디버깅이 어려웠습니다.
    - **해결:** `cron` 작업 등록 시 `* * * * * /path/to/script.sh 2>> /path/to/error.log` 와 같이 에러 리디렉션을 설정했습니다. 이를 통해 `cron` 환경에서 발생하는 모든 오류를 파일로 기록할 수 있었고, 로그를 보며 `jq` 패키지를 설치하거나, 명령어의 전체 경로를 명시하는 등 문제를 정확히 해결할 수 있었습니다.

- **문제 3: AI 분석 결과 통합 오류**
    - **현상:** 슬랙 메시지에 AI 분석 결과 대신 다른 정보(CPU 사용량)가 표시되었습니다.
    - **원인:** 스크립트 내에서 여러 정보를 처리하며 변수 할당 로직이 꼬여, 의도치 않은 값이 최종 메시지에 포함되었습니다.
    - **해결:** 각 정보(AI 분석, 자원 사용량)를 `AI_SUMMARY`, `RESOURCE_USAGE` 등 명확한 이름의 별도 변수에 저장한 뒤, 마지막에 이 변수들을 조합하여 최종 메시지를 만드는 방식으로 로직을 재구성하여 해결했습니다. 복잡한 작업을 할 때는 데이터 처리 단계를 명확히 분리하는 것의 중요성을 깨달았습니다.
