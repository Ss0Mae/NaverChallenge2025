1. **핵심 용어 이해하기**

위 도구들에서 본 용어들이 바로 가상 메모리의 핵심 요소입니다.

- **스왑 메모리 (Swap Memory)**: RAM이 부족할 때, 당장 사용하지 않는 메모리 데이터를 하드디스크의 특정 공간으로 옮겨두는 것을 '스왑 아웃'이라고 합니다. 이 공간이
바로 스왑 메모리(또는 스왑 파일)입니다. 나중에 데이터가 다시 필요해지면 디스크에서 RAM으로 가져옵니다('스왑 인'). 디스크는 RAM보다 훨씬 느리기 때문에 스왑이
자주 발생하면 시스템 성능이 저하됩니다.
- **공유 메모리 (Shared Memory)**: 여러 프로세스가 공통으로 사용하는 라이브러리나 프레임워크 같은 코드/데이터입니다. 예를 들어, macOS의 기본 라이브러리들은 한 번만
메모리에 로드되고 여러 앱이 이를 '공유'해서 사용합니다. 이렇게 함으로써 메모리를 크게 절약할 수 있습니다. top에서 보이는 SHMEM이 이와 관련됩니다.
- **압축된 메모리 (Compressed Memory)**: macOS의 기능입니다. 사용하지 않는 메모리를 무작정 스왑 아웃하기 전에, 먼저 CPU를 사용해 압축해서 공간을 확보합니다.
디스크에 접근하는 것보다 압축/해제하는 것이 훨씬 빠르기 때문에 성능에 유리합니다.

## 📚 프로세스별 메모리 분석 리서치

이제 특정 프로그램(프로세스) 하나가 메모리를 어떻게 사용하는지 더 깊게 들여다보겠습니다.

### 분석 도구 사용해보기

1. `ps aux`: 현재 실행 중인 모든 프로세스의 스냅샷을 보여줍니다. 특정 프로세스를 보려면 grep과 함께 사용합니다.
    - 실행 예시: ps aux | grep "Google Chrome"
    - 확인할 지표:
        - `VSZ` (Virtual Size): 프로세스가 요청한 총 가상 메모리 크기. 실제로 RAM을 사용하지 않더라도, 운영체제에 "이만큼의 주소 공간을 예약할게"라고 말한
        크기입니다.
        - `RSS` (Resident Set Size): 현재 실제 물리 메모리(RAM)에 올라가 있는 메모리의 크기. VSZ보다 훨씬 현실적인 메모리 사용량 지표입니다.

### 영역별 분석의 현실

요구사항에 나온 것처럼 "프로세스 메모리에서 Text, Heap, Stack 영역이 각각 몇 바이트를 차지하는지"를 ps나 top 같은 기본 명령어로 정확히 보는 것은 매우
어렵습니다. 이 정보는 보통 lldb 같은 디버거나 전문 프로파일링 도구를 사용해야 상세히 볼 수 있습니다.

하지만 우리는 이 개념을 이해하고 시뮬레이터를 만드는 것이 목표이므로, 각 영역의 역할을 아는 것이 더 중요합니다.

- Text: 우리가 작성한 코드(기계어)가 저장되는 곳. (읽기 전용)
- Stack: 함수가 호출될 때마다 지역 변수, 매개변수 등이 차곡차곡 쌓이는 공간. (LIFO 구조)
- Heap: 개발자가 원할 때 동적으로 메모리를 할당하고 해제하는 공간. (예: new, malloc)

## 📚 Node.js 프로그램의 힙(Heap) 메모리 분석

Node.js는 V8 엔진을 사용하며, 힙 메모리를 분석할 수 있는 도구를 내장하고 있습니다.

### 가장 간단한 방법: `process.memoryUsage()`

코드 내에서 현재 메모리 사용량을 직접 확인할 수 있습니다.

```jsx
test-memory.js
console.log(process.memoryUsage());
```

- `heapTotal`: V8 엔진이 힙을 위해 확보한 전체 메모리 크기
- `heapUsed`: 현재 사용 중인 힙 메모리 크기

```jsx
{
  rss: 39403520,
  heapTotal: 5341184,
  heapUsed: 4457320,
  external: 1475852,
  arrayBuffers: 10515
}
```

### 다른 방법: Chrome DevTools

Node.js 프로그램을 디버그 모드로 실행하여 Chrome 브라우저의 개발자 도구로 메모리 상태를 시각적으로 분석할 수 있습니다. 메모리 누수를 잡을 때 필수적인 기술입니다.

1. 실행: 터미널에 `node --inspect simulator.js` 와 같이 --inspect 플래그를 붙여 실행합니다.
2. 연결: Chrome 브라우저를 열고 주소창에 chrome://inspect 를 입력합니다. 
3. 분석: 개발자 도구 창이 뜨면 `Memory` 탭으로 이동합니다. 여기서 `Take snapshot` 버튼을 누르면 특정 시점의 힙 상태를 사진 찍듯이 저장하여 어떤 객체가 메모리를 얼마나 차지하는지 상세히 볼 수 있습니다.

## 📚 함수는 어떻게 호출되고 돌아오는가

- 함수 하나가 호출될 때마다, 그 함수만을 위한 독립적인 작업 공간이 스택(Stack) 에 생성됩니다. 이 공간을 바로 스택 프레임(Stack Frame) 이라고 부릅니다.

### 스택 프레임의 구성 요소

스택 프레임 안에는 보통 아래와 같은 정보들이 들어갑니다. 시뮬레이터에서도 이 정보들을 스택에 넣고 빼는 과정을 흉내 내야 합니다.

- 복귀 주소 (Return Address): 함수 실행이 끝나고 돌아가야 할 코드(TEXT 영역)의 주소. `CALL` 명령을 실행한 바로 다음 명령어의 주소입니다.
- 매개변수 (Parameters): 함수에 전달된 인자들.
- 지역 변수 (Local Variables): 함수 안에서 선언된 변수들. `VAR A: INT` 같은 명령어로 선언되는 변수들이 여기에 해당합니다.
- 이전 프레임 포인터 (Old Frame Pointer): 이 함수를 호출한 '이전' 함수의 스택 프레임 시작 주소를 가리키는 값. 스택 프레임들을 체인처럼 연결하여 `callstack()`같은 기능을 구현할 때 사용됩니다.

<aside>
✅ 어떤 메인함수에서 임의의 `foo` 함수를 호출한다고 생각을 해보자

</aside>

1. `CALL foo` 실행 전: main 함수의 스택 프레임만 스택에 존재합니다.
2. `CALL foo` 실행:
    - `foo`를 위한 스택 프레임 생성 시작
    - Push : foo가 끝나고 돌아와야 할 main의 주소(복귀 주소)를 스택에 push 합니다.
    - Push: main 함수의 스택 프레임 주소를 push 합니다 (이전 프레임 포인터).
    - Push: foo 함수의 지역 변수(`VAR K: INT`)들을 위한 공간을 스택에 push 합니다.
    - 이제 스택의 꼭대기에는 foo 함수의 스택 프레임이 완전히 만들어졌습니다.
3. `RETURN` 실행:
    - `foo`의 스택 프레임 해체
    - Pop (빼기): foo의 지역 변수 공간을 스택에서 pop 합니다.
    - Pop: 저장해뒀던 main 함수의 스택 프레임 주소를 pop 하여 복원합니다.
    - Pop: 저장해뒀던 복귀 주소를 pop 하여 PC(Program Counter)를 그 주소로 이동시킵니다.
    - 이제 프로그램은 main 함수로 무사히 복귀했고, 스택은 foo가 호출되기 이전 상태로 깨끗하게 정리되었습니다.

## 📚 필요할 때마다 메모리를 빌리고 반납하기 (힙과 동적 할당)

> 스택은 함수가 시작되고 끝날 때 알아서 메모리가 정리되는 '자동' 영역입니다. 하지만 프로그램이 실행되는 도중에, 우리가 원할 때 원하는 크기만큼 메모리를
할당하고 싶다면 어떻게 할까
> 

### 스택과 힙의 차이점

- 스택 (Stack):
    - 관리 주체: 컴파일러/언어 런타임 (자동)
    - 생존 기간: 함수 호출이 끝나면 사라짐
    - 용도: 지역 변수, 매개변수 등 크기가 정해진 데이터
    - 장점: 매우 빠름 (포인터 이동만으로 할당/해제)
    - 단점: 유연성이 떨어지고 크기 제한이 있음
- 힙 (Heap):
    - 관리 주체: 프로그래머 (수동)
    - 생존 기간: 프로그래머가 명시적으로 해제하기 전까지 계속 유지됨
    - 용도: 프로그램 실행 중 크기가 변하거나, 함수 호출과 상관없이 오래 유지되어야 하는 데이터 (객체, 배열 등)
    - 장점: 필요에 따라 원하는 크기의 메모리를 동적으로 할당 가능
    - 단점: 스택보다 느리고, 메모리 누수(Memory Leak)의 위험이 있음

### alloc과 free

**우리가 구현해야할 시뮬레이터의 `alloc`과 `free` 함수는 `C언어의 malloc / free나 C++의 new / delete`와 같은 역할을 합니다**

- `alloc` (할당):
    - “힙 공간아, 나 INT 타입 10개 들어갈 만한 공간 좀 빌려줘!" 라고 요청하는 것과 같습니다.
    - 힙 관리자는 비어있는 공간을 찾아서 요청한 크기만큼 할당해주고, 그 공간의 시작 주소(포인터)를 돌려줍니다.
    - 이 주소(포인터)를 잃어버리면, 할당된 힙 메모리에 다시는 접근할 수도, 해제할 수도 없게 됩니다. (메모리 누수의 원인)
    - 그래서 시뮬레이터에서는 alloc이 반환한 힙 주소를 스택에 있는 변수에 저장하여 관리하려합니다.
- `free` (해제):
    - "아까 빌렸던 이 주소의 공간, 이제 다 썼으니 반납할게!" 라고 알리는 것입니다.
    - 힙 관리자는 해당 공간을 '사용 가능' 상태로 만들어서 나중에 다른 alloc 요청이 오면 재사용할 수 있도록 합니다.

## Memory Fragmentation

> 메모리 단편화는 컴퓨터의 메모리 공간이 여러 조각으로 나뉘어, 실제 사용 가능한 메모리의 총합은 충분하지만 연속된 큰 공간이 없어 새로운 할당을 만족하지 못하는 현상을 말한다
> 

### 내부 단편화

개념:
메모리를 할당할 때, 요청된 크기보다 더 큰 공간을 할당하게 되어 남는 공간을 의미합니다. 즉, 할당된 블록의 내부에서 사용되지 않고 낭비되는 공간입니다.

예시:

- 운영체제가 메모리를 4KB 크기의 페이지(Page) 단위로 관리한다고 가정을 하면.
- 만약 10KB 크기의 프로세스를 할당해야 한다면, 총 3개의 페이지(4KB * 3 = 12KB)를 할당해야 합니다.
- 이때 프로세스는 10KB만 사용하고, 나머지 2KB는 할당은 되었지만 사용되지 않는 내부 단편화가 발생합니다

### 외부 단편화

개념:
메모리 할당과 해제가 반복되면서, 할당된 공간들 사이에 작은 여유 공간(Hole)들이 생겨나는 현상입니다. 이 작은 공간들의 총합은 크지만, 서로 연속적이지 않아 큰 메모리 블록을 필요로 하는 요청을 처리할 수 없게 됩니다. 즉, 할당된 블록들의 외부에 낭비되는 공간입니다.

예시:

- 100MB의 메모리가 있고, 3개의 프로세스 A(20MB), B(30MB), C(25MB)가 순서대로 할당되었다고 가정합니다.
- [ A(20) | B(30) | C(25) | --- (25MB 남음) --- ]
- 여기서 프로세스 B가 작업을 마치고 메모리에서 해제됩니다.
- [ A(20) | --- (30MB 비어있음) --- | C(25) | --- (25MB 남음) --- ]
- 이제 총 55MB의 여유 공간이 있지만, 두 공간이 분리되어 있습니다. 만약 40MB 크기의 새로운 프로세스 D를 할당하려고 하면, 연속된 40MB 공간이 없으므로 할당에 실패합니다. 이것이 외부 단편화입니다.

## 📚 가비지 컬렉션의 기본 개념

컴퓨터 프로그램은 실행될 때 데이터를 저장하기 위해 메모리라는 공간을 사용합니다. 이 메모리는 한정된 자원이기 때문에, 더 이상 사용하지 않는 데이터는 메모리에서 지워서 공간을 확보해야 합니다. 그렇지 않으면 메모리가 꽉 차서 프로그램이 멈추는 메모리 누수(Memory Leak) 현상이 발생할 수 있습니다.

과거 C/C++ 같은 언어에서는 개발자가 malloc()으로 메모리를 직접 할당하고, free()로 직접 해제해주어야 했습니다. 이는 매우 번거롭고, 깜빡하고 해제하지 않으면 바로 메모리 누수로 이어졌습니다

가비지 컬렉션(Garbage Collection, GC) 은 바로 이 문제를 해결하기 위해 등장했습니다. 개발자를 대신하여 더 이상 사용되지 않는 메모리(Garbage) 를 알아서 찾아내고 해제해주는 자동 메모리 관리 기법입니다.

## 대표적인 가비지 컬렉션 알고리즘

가비지 컬렉터가 "쓰레기"를 찾아내는 방법에는 여러 가지가 있지만, 가장 대표적인 두 가지는 **Reference Counting(참조 카운팅)과 Mark-and-Sweep(표시하고 쓸기)** 입니다.

### Reference Counting (참조 카운팅)

가장 간단한 알고리즘 중 하나입니다. 어떤 객체(데이터)를 참조하는 변수가 몇 개인지 세는 방식입니다

- 동작 방식:
    1. 객체가 생성되고 변수에 할당되면, 해당 객체의 참조 카운트가 1이 됩니다.
    2. 다른 변수가 같은 객체를 참조하면, 카운트가 1 증가합니다.
    3. 객체를 참조하던 변수가 다른 값을 참조하게 되거나 사라지면, 카운트가 1 감소합니다.
    4. 참조 카운트가 0이 되면, 아무도 그 객체를 사용하지 않는다는 의미이므로 "쓰레기"로 간주하고 즉시 메모리에서 해제합니다.
- 장점:
    - 쓰레기가 발생하는 즉시 메모리를 해제하므로, 프로그램이 멈추는 현상(stop-the-world)이 거의 없습니다.
- 치명적인 단점: 순환 참조 (Circular Reference)
    - 두 개 이상의 객체가 서로를 참조하는 경우, 참조 카운트가 절대 0이 되지 않아 메모리 누수가 발생합니다.

```jsx
// 순환 참조 예시
  function createCircularReference() {
    let objA = {};
    let objB = {};

    objA.b = objB; // objA가 objB를 참조 (objB의 참조 카운트: 1)
    objB.a = objA; // objB가 objA를 참조 (objA의 참조 카운트: 1)

   // 함수가 끝나면 objA와 objB 변수는 사라지지만,
   // 두 객체는 서로를 참조하고 있어 참조 카운트가 0이 되지 않는다.
   // 따라서 메모리에서 해제되지 않는다.
  }
createCircularReference();
```

### Mark-and-Sweep (표시하고 쓸기)

현재 대부분의 모던 프로그래밍 언어(자바스크립트 포함)가 사용하는 방식이며, 순환 참조 문제를 해결했습니다.

- 동작 방식:
    1. Mark (표시) 단계:
        - 가비지 컬렉터는 루트(Root)에서부터 시작하여, 참조할 수 있는 모든 객체를 찾아 방문하고 "살아있음(reachable)" 표시를 합니다.
        - (루트는 전역 변수, 현재 함수의 지역 변수 등 코드에서 직접 접근 가능한 시작점을 의미합니다.)
    2. Sweep (쓸기) 단계:
        - 전체 메모리를 순회하면서, "살아있음" 표시가 없는 모든 객체를 "쓰레기"로 간주하고 메모리에서 해제합니다.
- 장점:
    - 순환 참조 문제를 완벽하게 해결합니다. 위 예시에서 objA와 objB는 루트에서 접근할 수 없으므로 "살아있음" 표시를 받지 못하고, 결국 수거됩니다.
- 단점:
    - GC가 동작하는 동안에는 프로그램 실행이 일시적으로 멈추는 "Stop-the-world" 현상이 발생할 수 있습니다. (물론 최신 GC는 이를 최소화하는 방향으로 발전했습니다.)

## 자바스크립트 (V8 엔진)의 가비지 컬렉션

Chrome 브라우저와 Node.js에서 사용하는 V8 엔진은 앞서 설명한 Mark-and-Sweep을 기본으로 사용하지만, 훨씬 더 정교하고 최적화된 방식을 사용합니다. 바로 세대별 가비지 컬렉션(Generational Garbage Collection) 입니다.

이 방식은 다음과 같은 통계적 사실에 기반합니다.

> "대부분의 객체는 생성된 지 얼마 안 되어 쓰레기가 된다."
> 

이 가설에 따라, V8은 객체를 '나이'에 따라 두 세대로 나누어 관리합니다.

1. **Young Generation (신생 세대)**
- 역할: 새로 생성된 모든 객체가 할당되는 곳입니다.
- 특징:
    - 메모리 공간이 작고, 가비지 컬렉션이 매우 빈번하게 발생합니다. **(이를 Minor GC 또는 Scavenge라고 부릅니다.)**
    - 이 영역은 다시 두 개의 동일한 크기 공간**(From-space, To-space)**으로 나뉩니다.
- 동작 방식 (Scavenge):
    1. 새 객체는 **From-space**에 할당됩니다.
    2. **From-space**가 꽉 차면 **Minor GC**가 실행됩니다.
    3. 살아남은 객체(여전히 접근 가능한 객체)를 **To-space**로 복사합니다.
    4. 살아남은 객체 복사가 끝나면, **From-spac**e에 남아있는 모든 것은 쓰레기이므로, **From-space** 전체를 깨끗하게 비웁니다.
    5. 마지막으로 **From-space**와 **To-space**의 역할을 바꿉니다. (이제 **To-space**가 **From-space**가 됩니다.)
    6. 이 과정을 한 번 겪고도 살아남은 객체는 "나이가 많은" 객체로 간주되어 **Old Generation**으로 이동(Promotion)됩니다.
- 이 방식은 살아있는 객체만 복사하고 나머지 공간을 통째로 비우기 때문에 매우 빠릅니다.
1. **Old Generation (기성 세대)**
- 역할: Young Generation에서 살아남아 넘어온, "오래된" 객체들이 머무는 곳입니다.
- 특징:
    - Young Generation보다 메모리 공간이 훨씬 크고, 가비지 컬렉션이 드물게 발생합니다. (이를 Major GC라고 부릅니다.)
    - Major GC는 상대적으로 속도가 느리고 더 많은 비용을 소모합니다.
- 동작 방식 (**Mark-Sweep-Compact**):
    1. Mark-Sweep: 기본적으로 Mark-and-Sweep 알고리즘을 사용하여 살아있는 객체를 표시하고, 표시되지 않은 쓰레기 객체들의 메모리를 해제합니다.
    2. **Mark-Compact**: Mark-Sweep을 반복하다 보면 메모리 공간 중간중간에 구멍이 뚫리는 파편화(Fragmentation) 현상이 발생합니다. 이를 해결하기 위해, 가끔씩
    살아있는 객체들을 한쪽으로 쭉 옮겨서 메모리를 압축하고 파편화된 공간을 없애는 **Mark-Compact** 과정을 수행합니다.

## 가비지 컬렉션과 성능

가비지 컬렉션은 자동 메모리 관리라는 엄청난 편의성을 제공하지만, . GC는 그 자체로 비용이 드는 작업이며, 프로그램 성능에 직접적인 영향을 줄 수 있습니다.

### "Stop-the-World"와 성능 저하

가장 큰 성능 문제는 앞서 언급된 "Stop-the-World" 입니다. GC가 실행되는 동안에는 자바스크립트 코드 실행을 포함한 모든 프로그램 동작이 일시적으로 멈춥니다.

- 사용자 경험(UX)에 미치는 영향:
    - **웹 프론트엔드**: GC가 실행되는 순간, 애니메이션이 뚝 끊기거나(Jank), 스크롤이 멈추거나, 버튼 클릭에 반응하지 않는 등 사용자는 "버벅거림"을 느끼게 됩니다. 특히 Minor GC는 짧지만 빈번하고, Major GC는 길고 강력한 멈춤을 유발할 수 있습니다.
    - **백엔드** (Node.js): 서버가 GC 때문에 멈추면, 들어오는 요청에 대한 응답 시간이 길어지거나(Latency 증가) 심한 경우 요청을 처리하지 못할 수도 있습니다.

V8 엔진은 이러한 멈춤 시간을 최소화하기 위해 병렬(Parallel) GC, 증분(Incremental) GC, 유휴 시간(Idle-time) GC 등 다양한 최적화 기법을 사용하지만, 멈춤을완벽하게 없앨 수는 없습니다.

### 개발자가 고려해야 할 점 (메모리 누수 방지)

가비지 컬렉터는 완벽하지 않습니다. 개발자의 코딩 습관에 따라 "쓰레기"임에도 불구하고 메모리에서 해제되지 않는 메모리 누수(Memory Leak) 가 발생할 수 있습니다. **GC는 "참조할 수 없는 객체"를 지우는 것이지, "필요 없는 객체"를 지우는 것이 아니기 때문입니다.**

다음은 메모리 누수를 유발하는 대표적인 안티 패턴입니다.

- 전역 변수 남용: 전역 변수는 애플리케이션이 종료될 때까지 계속 살아있으므로, 거대한 객체를 전역 변수에 할당하고 잊어버리면 메모리 누수로 이어집니다.

```jsx
 function processData() {
   // 실수로 window.hugeData에 할당하면 전역 변수가 됨
   hugeData = new Array(1000000).fill('some data');
 }
 processData();
 // hugeData는 더 이상 필요 없지만, 전역에 남아있어 GC되지 않음
```

- 분리된 DOM 노드: DOM에서 특정 요소를 제거했지만, 자바스크립트 코드 어딘가에서 여전히 그 DOM 노드를 참조하고 있는 경우입니다. DOM 노드는 메모리에서 해제되지
않습니다.

```jsx
 const elements = [];
 const myDiv = document.getElementById('myDiv');
 elements.push(myDiv);

 // DOM에서는 제거했지만...
 document.body.removeChild(myDiv);

 // elements 배열이 myDiv를 계속 참조하고 있어 GC되지 않음
 console.log(elements[0] === myDiv); // true
```

- 해제되지 않은 타이머/이벤트 리스너: `setInterval`이나 `addEventListener`를 사용한 후, 더 이상 필요 없을 때 `clearInterval`이나 `removeEventListener`로 해제하지
않으면, 콜백 함수와 콜백이 참조하는 모든 것이 메모리에 계속 남게 됩니다.

## 🗺️ 지도: 학습 내용 점검 및 확장

### 1) 스스로 확인할 사항

> 챌린지 동안 문제 해결 과정 중 학습한 내용에 대해 스스로 점검해야 할 내용입니다. 학습 배경 지식과 함께 문제 해결 과정에서의 경험을 정리해 봅시다.

#### **Q1. 구현한 시뮬레이터의 각 함수는 어떻게 테스트했는가?**

시뮬레이터의 정확성을 검증하기 위해 각 기능별로 명확한 시나리오를 설정하고, 예상되는 메모리 상태와 실제 결과를 비교하는 방식으로 테스트를 진행했습니다.

-   **`locate`, `setSize`:** 초기 설정 함수들은 `reset`으로 상태를 초기화한 후, 여러 타입과 함수를 등록하고 내부 `Map` 객체에 값이 정상적으로 저장되었는지 확인했습니다.
-   **`VAR`, `SET`, `RELEASE`:** `main` 함수에 간단한 변수 할당(`VAR A: INT`), 값 변경(`SET A = 10`), 해제(`RELEASE A`) 코드를 넣고 `다음()` 명령어로 한 단계씩 실행했습니다. 각 단계마다 `heapdump()`와 `usage()`를 호출하여 힙 메모리가 예상대로 할당/변경되고, 스택 포인터가 올바른 힙 주소를 가리키는지 검증했습니다.
-   **`CALL`, `RETURN`:** `main`에서 `foo` 함수를 호출하고, `foo`가 값을 반환하는 시나리오를 사용했습니다.
    -   `CALL` 직후: `callstack()`을 호출하여 `main -> foo` 호출 스택이 정상적으로 쌓였는지, `PC`가 `foo`의 시작 주소로 이동했는지 확인했습니다.
    -   `RETURN` 직후: `$RETURN` 레지스터에 반환 값이 저장되었는지, `callstack()`에서 `foo`가 사라졌는지, `PC`가 `main`의 복귀 주소로 돌아왔는지 확인했습니다.
-   **`garbageCollect` (미구현):** 만약 구현했다면, 의도적으로 힙 객체에 대한 참조를 스택에서 제거한 뒤 `garbageCollect()`를 호출하고, `heapdump()`를 통해 해당 힙 객체가 'free' 상태로 변경되었는지 확인하는 방식으로 테스트했을 것입니다.

#### **Q2. 일반적인 프로세스 메모리 모델은 어떻게 구성되는가?**

프로세스가 운영체제로부터 할당받는 메모리 공간은 논리적으로 다음과 같은 주요 영역으로 나뉩니다.

1.  **Text (Code) 영역:** 실행할 프로그램의 기계어 코드가 저장되는 공간입니다. CPU는 이 영역의 명령어들을 순차적으로 읽어 실행합니다. 데이터가 아닌 코드 자체이므로, 프로그램 실행 중에 변경될 일이 없도록 **읽기 전용(Read-only)**으로 설정됩니다.
2.  **Data 영역:** 전역 변수(Global variables)와 정적 변수(Static variables)가 저장되는 공간입니다.
    -   **초기화된 데이터 (Initialized Data):** `int a = 10;` 처럼 초기값이 있는 변수들이 저장됩니다.
    -   **초기화되지 않은 데이터 (Uninitialized Data, BSS):** `int b;` 처럼 초기값이 없는 변수들이 저장되며, 프로그램 시작 시 모두 0으로 초기화됩니다.
3.  **Heap 영역:** 프로그래머가 직접 메모리를 할당하고 해제하는 동적 메모리 공간입니다. `malloc`, `new` 등으로 할당하며, 낮은 주소에서 높은 주소 방향으로 자라납니다. 런타임에 크기가 결정되는 데이터(객체, 배열 등)를 저장하는 데 사용됩니다.
4.  **Stack 영역:** 함수 호출과 관련된 데이터를 임시로 저장하는 공간입니다. 함수의 매개변수, 지역 변수, 복귀 주소 등이 저장됩니다. 높은 주소에서 낮은 주소 방향으로 자라나며, LIFO(Last-In, First-Out) 구조를 가집니다. 컴파일 타임에 크기가 결정되며, 매우 빠른 속도로 메모리를 할당하고 해제할 수 있습니다.

#### **Q3. Node.js의 메모리 모델은 일반 모델과 어떤 차이가 있는가?**

Node.js는 V8 자바스크립트 엔진 위에서 동작하므로, 전통적인 C/C++ 기반 프로세스 메모리 모델과는 약간 다른 관점에서 메모리를 관리합니다.

-   **공통점:** Node.js 프로세스도 운영체제로부터 Text, Stack, Heap 등의 메모리 영역을 할당받는다는 근본적인 구조는 동일합니다.
-   **차이점 (V8 엔진의 관점):**
    -   **개발자의 직접 제어 부재:** C/C++과 달리, 개발자가 `malloc`이나 `free`처럼 직접 메모리 주소를 다루며 힙을 제어하지 않습니다. `let obj = {}` 와 같이 객체를 생성하면 V8 엔진이 알아서 힙에 메모리를 할당합니다.
    -   **가비지 컬렉션:** 가장 큰 차이점입니다. V8 엔진은 **가비지 컬렉터(GC)**를 내장하여, 더 이상 참조되지 않는 객체를 자동으로 찾아 메모리에서 해제합니다. 이로 인해 개발자는 메모리 관리에 대한 부담을 크게 덜 수 있지만, GC 동작 원리를 이해하지 못하면 메모리 누수를 유발할 수 있습니다.
    -   **힙의 세분화:** V8의 힙은 단순히 하나의 큰 덩어리가 아니라, **New Space(Young Generation)**와 **Old Space(Old Generation)** 등으로 나뉘어 세대별 가설에 기반한 효율적인 GC를 수행합니다.

#### **Q4. 리눅스(macOS)의 가상 메모리 관리 방식은?**

리눅스나 macOS와 같은 현대 운영체제는 **가상 메모리(Virtual Memory)** 기법을 사용하여 물리 메모리(RAM)를 효율적이고 안전하게 관리합니다.

-   **핵심 개념:** 각 프로세스에게 실제 물리 메모리 주소가 아닌, 독립된 가상 주소 공간(Virtual Address Space)을 할당합니다. 이 가상 주소 공간은 0부터 시작하는 연속적인 메모리처럼 보이지만, 실제로는 물리 메모리의 여러 위치에 흩어져 있거나 디스크에 저장될 수 있습니다.
-   **MMU (Memory Management Unit):** CPU와 물리 메모리 사이에 위치한 하드웨어 장치로, 가상 주소를 실제 물리 주소로 변환하는 역할을 담당합니다.
-   **페이징 (Paging):** 가상 메모리와 물리 메모리를 일정한 크기의 블록(보통 4KB)으로 나누어 관리합니다. 가상 메모리의 블록을 **페이지(Page)**, 물리 메모리의 블록을 **프레임(Frame)**이라고 부릅니다. MMU는 **페이지 테이블(Page Table)**을 참조하여 어떤 페이지가 어떤 프레임에 매핑되는지 관리합니다.
-   **장점:**
    -   **메모리 보호:** 각 프로세스는 자신만의 독립된 주소 공간을 가지므로, 다른 프로세스의 메모리에 접근할 수 없어 시스템이 안정적으로 유지됩니다.
    -   **효율적인 메모리 사용:** 프로그램의 일부만 물리 메모리에 올려놓고 실행할 수 있으며(Demand Paging), 여러 프로세스가 동일한 라이브러리를 공유(Shared Memory)하여 물리 메모리를 절약할 수 있습니다.
    -   **물리 메모리보다 큰 공간 사용:** 당장 필요 없는 데이터는 디스크(스왑 공간)에 내려놓고, 필요할 때 다시 RAM으로 가져오는 **스와핑(Swapping)**을 통해 실제 RAM 크기보다 더 큰 프로그램을 실행할 수 있습니다.

#### **Q5. 스택과 힙 방식의 비교**

| 특징 | 스택 (Stack) | 힙 (Heap) |
| :--- | :--- | :--- |
| **관리 주체** | 컴파일러/런타임 (자동) | 프로그래머/GC (수동/자동) |
| **할당/해제 속도** | 매우 빠름 (포인터 이동) | 상대적으로 느림 (탐색/관리 오버헤드) |
| **생존 기간** | 함수 호출 동안 (블록 스코프) | 명시적으로 해제하거나 GC될 때까지 |
| **데이터 구조** | LIFO (Last-In, First-Out) | 규칙 없음 (복잡한 자료구조) |
| **크기** | 작고, 컴파일 시 결정 | 크고, 런타임 시 동적 변경 |
| **주요 용도** | 지역 변수, 매개변수, 복귀 주소 | 객체, 배열, 전역적으로 유지될 데이터 |
| **단편화** | 발생하지 않음 | 내부/외부 단편화 발생 가능 |

### 2) 다같이 확인할 사항

> 챌린지 문제 해결 과정 이후 동료와 서로 다른 시각에서 설명하고, 이해한 내용을 공유해야 할 내용입니다. 그러기 위해서 배경 지식과 본인의 의견을 함께 정리해 봅시다.

#### **Q1. 언어/플랫폼별 메모리 관리(GC) 방식의 차이점은?**

-   **C/C++ (플랫폼 네이티브):** GC가 없습니다. 개발자가 `malloc/free`, `new/delete`를 통해 모든 메모리를 직접 관리해야 합니다. 최고의 성능을 낼 수 있지만, 메모리 누수(leak), 이중 해제(double free) 등 치명적인 버그가 발생하기 쉽습니다.
-   **Java (JVM):** JVM의 GC는 매우 발전된 형태로, V8과 유사하게 세대별 GC(Generational GC)를 사용합니다. 다양한 GC 알고리즘(Serial, Parallel, G1, ZGC 등)을 제공하며, 애플리케이션의 특성에 맞게 GC를 튜닝할 수 있는 옵션이 풍부합니다. 주로 서버 환경에서 안정성과 처리량에 중점을 둡니다.
-   **JavaScript (V8):** 앞서 설명한 세대별 GC를 사용합니다. 주로 브라우저나 UI 환경에서 사용되므로, 사용자 인터랙션에 방해를 주지 않도록 'Stop-the-World' 시간을 최소화하는 데 중점을 둡니다.
-   **Python (CPython):** 주된 GC 메커니즘으로 **참조 카운팅(Reference Counting)**을 사용합니다. 객체의 참조 횟수가 0이 되면 즉시 메모리를 해제하여 GC로 인한 멈춤이 거의 없습니다. 하지만 순환 참조 문제를 해결하기 위해 부가적으로 세대별 GC(Mark-and-Sweep)를 보조적으로 사용합니다.
-   **Swift/Objective-C (Apple):** **ARC(Automatic Reference Counting)**를 사용합니다. 컴파일 타임에 컴파일러가 코드를 분석하여, 객체의 참조 카운트를 관리하는 코드를 자동으로 삽입해 줍니다. 런타임 오버헤드가 적다는 장점이 있지만, 순환 참조 문제는 개발자가 `weak`나 `unowned` 같은 키워드를 사용해 직접 해결해야 합니다.

#### **Q2. 메모리 관리를 위한 자료구조 선택의 어려움**

이번 시뮬레이터 구현에서 힙 메모리를 관리하는 방식은 명시되지 않았습니다. 만약 실제 `malloc`을 구현한다면, 어떤 자료구조로 '사용 가능한 메모리 블록(free list)'을 관리할지 결정해야 합니다.

-   **단순 연결 리스트 (Singly Linked List):** 가장 간단한 방법입니다. 해제된 메모리 블록들을 연결 리스트로 쭈욱 엮어놓는 것입니다.
    -   **장점:** 구현이 매우 쉽습니다.
    -   **어려움:** 메모리 할당 요청이 들어올 때마다 리스트 전체를 순회하며 적절한 크기의 블록을 찾아야 하므로 비효율적입니다. (First-fit, Best-fit, Worst-fit 등의 탐색 전략 필요)
-   **이진 탐색 트리 (Binary Search Tree):** 메모리 블록의 크기를 기준으로 트리를 구성하면, 적절한 크기의 블록을 O(log N) 시간 복잡도로 빠르게 찾을 수 있습니다.
    -   **장점:** 탐색 속도가 빠릅니다.
    -   **어려움:** 할당과 해제가 반복되면서 트리가 한쪽으로 치우쳐져 성능이 저하될 수 있습니다. 이를 방지하기 위해 균형 잡힌 트리(e.g., Red-Black Tree)를 사용해야 하는데, 구현이 매우 복잡해집니다.
-   **버디 메모리 할당 (Buddy System):** 리눅스 커널에서 사용하는 기법 중 하나입니다. 메모리를 2의 거듭제곱 크기로 나누고, 같은 크기의 인접한(buddy) 블록들을 관리하는 방식입니다.
    -   **장점:** 특정 크기의 블록을 찾기 쉽고, 해제 시 인접한 버디 블록과 쉽게 합칠 수 있어 외부 단편화 문제를 완화하는 데 효과적입니다.
    -   **어려움:** 2의 거듭제곱 크기로만 할당하므로, 요청된 크기와 할당된 크기 사이의 차이로 인한 내부 단편화가 발생하기 쉽습니다.

#### **Q3. 힙 메모리 단편화(Fragmentation)와 효율적인 관리 방안**

힙 메모리를 할당하고 해제하는 과정이 반복되면, 메모리 공간 중간중간에 사용하지 못하는 작은 빈 공간들이 생겨납니다. 이를 **단편화**라고 합니다.

-   **내부 단편화:** 할당된 메모리 블록이 실제 요청한 크기보다 커서, 블록 내부에 사용되지 않고 낭비되는 공간. (예: 10바이트를 요청했는데 16바이트 블록을 할당)
-   **외부 단편화:** 사용 가능한 메모리 총량은 충분하지만, 모두 작은 조각으로 흩어져 있어 큰 메모리 요청을 처리하지 못하는 상태.

`malloc`과 같은 메모리 할당자는 이러한 단편화를 최소화하기 위해 정교한 전략을 사용합니다.

-   **탐색 전략 (Placement Strategy):**
    -   **First-fit:** 가용 공간 리스트에서 가장 먼저 발견되는 충분한 크기의 블록을 할당. 빠르지만, 리스트 앞쪽에 작은 조각들이 쌓일 수 있음.
    -   **Next-fit:** 마지막으로 할당된 위치부터 탐색을 시작. First-fit보다 빠를 수 있지만, 전체적인 메모리 사용률이 나빠질 수 있음.
    -   **Best-fit:** 요청한 크기와 가장 비슷한 크기의 블록을 할당. 메모리 낭비는 적지만, 모든 가용 공간을 탐색해야 하므로 느리고 아주 작은 조각들이 남을 수 있음.
-   **분리 가용 리스트 (Segregated Free Lists):** 모든 가용 블록을 하나의 리스트로 관리하는 대신, 특정 크기(e.g., 8, 16, 32바이트...)별로 여러 개의 리스트를 두는 방식입니다.
    -   **장점:** 작은 크기의 객체를 요청할 때, 해당 크기의 리스트에서 바로 블록을 가져올 수 있어 매우 빠르고 단편화도 적습니다. `malloc`의 현대적인 구현체들은 대부분 이 방식을 기반으로 합니다.
-   **슬랩 할당 (Slab Allocation):** 리눅스 커널에서 자주 사용되는 기법으로, 분리 가용 리스트와 유사합니다. 자주 사용되는 동일한 크기의 객체들(e.g., 프로세스 디스크립터)을 위해 미리 '슬랩(Slab)'이라는 캐시를 만들어 둡니다. 객체를 생성하고 파괴할 때, 이 캐시를 재사용함으로써 초기화/파괴 오버헤드를 줄이고 단편화를 방지합니다.

이러한 기법들은 결국 **"자주 사용하는 크기의 메모리는 미리 준비해두고(caching), 비슷한 크기의 요청끼리 모아서 처리하여(pooling) 낭비를 줄인다"**는 공통적인 아이디어를 가집니다.
