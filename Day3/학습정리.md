# Day 3: 학습 정리 (피어 세션 최종본)


## 🧐 1. 스스로 확인할 사항 (개인 학습 점검)

### 1.1. 컴파일러 이론과 실제 구현: Tokenizer, Lexer, Parser

**A. 컴파일러 이론에서의 역할**

-   **렉서(Lexer) / 토크나이저(Tokenizer):** 소스 코드를 문자 스트림으로 읽어들여, 문법적으로 의미 있는 최소 단위인 토큰(Token)으로 분해합니다. 공백이나 주석을 제거하는 역할도 수행합니다.
-   **파서(Parser):** 렉서로부터 전달받은 토큰 스트림을 기반으로, 해당 언어의 문법 규칙에 따라 토큰들의 관계를 분석하여 추상 구문 트리(AST, Abstract Syntax Tree)를 생성합니다.

**B. 내 프로그램에서의 역할**

-   **`_tokenize` (렉서/토크나이저):** 이 메서드는 XML 문자열을 입력받아 `<tag>`, `</tag>`, `text` 등 의미 있는 조각으로 분해하여 토큰 객체의 배열을 생성했습니다. 이는 컴파일러의 렉서와 동일한 역할을 수행합니다.
-   **`_buildTree` (파서):** 이 메서드는 `_tokenize`가 생성한 토큰 배열을 입력받아, 여는 태그와 닫는 태그의 관계를 해석하며 중첩된 트리 구조를 만들었습니다. 이는 토큰의 관계를 분석해 AST를 만드는 파서의 역할에 해당합니다.

### 1.2. 트리(Tree) 자료구조와 재귀(Recursion)를 이용한 문제 해결

-   **트리 구조의 적합성:** XML/HTML의 중첩 구조는 계층적 데이터를 표현하는 데 최적화된 트리 자료구조로 표현하기에 매우 적합합니다. 각 엘리먼트는 노드(Node)가 되고, `children` 프로퍼티를 통해 자식 노드들을 참조하며 전체 구조를 형성했습니다.

-   **재귀를 통한 트리 생성 및 순회:** 트리는 본질적으로 재귀적인 자료구조이므로, 트리를 생성하거나 탐색하는 데 재귀는 필수부가결 합니다.
    -   **`_buildTree`의 재귀 설계:** `tag_open` 토큰을 만나면 새 노드를 만들고, 그 자식을 처리하기 위해 자기 자신을 재귀 호출했습니다. 짝이 맞는 `tag_close` 토큰을 만나면 재귀를 종료(`Base Case`)하는 방식으로 복잡한 중첩 구조를 효과적으로 처리했습니다.
    -   **DFS 순회:** `elementByAttribute`와 같은 유틸리티 함수들은 깊이 우선 탐색(DFS) 방식으로 트리를 순회합니다. 특정 노드의 자식들을 먼저 끝까지 탐색하는 이 방식은 재귀를 통해 구현되었습니다.

### 1.3. 웹 브라우저 DOM 파서와 내 파서 비교

| 구분 | 내가 구현한 파서 | 웹 브라우저 DOM |
| :--- | :--- | :--- |
| **기본 구조** | `{ name, attributes, children, value }` 형태의 단순 JavaScript 객체 | `Node` 인터페이스를 상속받는 복잡한 객체. `nodeType`, `parentNode`, `childNodes` 등 수많은 프로퍼티와 `getElementById` 같은 메서드를 가짐. |
| **오류 처리** | 문법에 맞지 않으면 파싱이 실패하거나 예기치 않은 결과를 낼 수 있음 (오류에 취약) | 닫는 태그가 없거나 순서가 틀려도 자체 규칙에 따라 최대한 복구하여 트리를 생성 (오류에 관용적) |
| **API** | `findXPath` 등 직접 구현한 몇 개의 탐색 함수만 존재 | `querySelector`, `getElementsByTagName` 등 강력하고 표준화된 탐색 및 조작 API를 풍부하게 제공 |

-   **브라우저 개발자 도구 활용:** Elements 탭에서 브라우저가 생성한 DOM 트리를 시각적으로 확인하고, 특정 엘리먼트의 XPath를 복사하여 `findXPath` 기능의 목표와 원리를 명확히 이해할 수 있었습니다.

### 1.4. 정규표현식(RegEx)의 활용

-   **정의:** 문자열에서 특정 패턴을 검색, 매칭, 치환하기 위해 사용하는 형식 언어입니다.
-   **구현에서의 활용:** `_parseAttributes` 메서드에서 `([\w:-]+)\s*=\s*"([^"]*)"` 라는 정규표현식을 사용하여 `name="value"` 형태의 속성을 효과적으로 추출했습니다.
-   **개선 가능성:** `_tokenize` 메서드의 복잡한 조건 분기들을 더 정교한 정규표현식으로 대체하여 코드를 간결하게 만들 수 있지만, 과도하게 복잡한 정규식은 오히려 가독성을 해칠 수 있어 장단점을 고려해야 합니다.

### 1.5. 구현 과정에서의 트러블슈팅 경험

-   **문제 1: 프롤로그(`<?xml ...>`) 파싱 오류**
    -   **원인:** `xml.startsWith("", i + 1)` 라는 잘못된 조건문이 주석 외 모든 태그를 프롤로그로 잘못 인식시켰습니다.
    -   **해결:** 프롤로그의 명확한 시작 패턴인 `<?`를 감지하도록 `xml.startsWith("?", i + 1)`로 조건을 수정하여 해결했습니다.

-   **문제 2: 리팩토링 중 재귀 로직 파괴**
    -   **원인:** `_buildTree` 메서드를 분리하며, 클로저를 통해 모든 재귀 호출이 공유하던 `tokenIndex` 상태 관리 매커니즘을 간과했습니다. 이로 인해 상태 동기화가 깨져 트리 구조가 망가졌습니다.
    -   **해결:** 상태 공유의 중요성을 인지하고, 안정적으로 동작하던 이전 버전의 단일 함수 로직으로 복원했습니다. 복잡한 상태를 공유하는 재귀 함수는 리팩토링 시 매우 신중해야 함을 배웠습니다.

-   **문제 3: Mermaid 다이어그램 텍스트 소실**
    -   **원인:** Mermaid 렌더러가 도형 안의 특수문자 `<`, `>`를 문법으로 오인하여 텍스트를 렌더링하지 못했습니다.
    -   **해결:** `<`는 `&lt;`, `>`는 `&gt;`와 같은 HTML 엔티티 코드로 치환하여 문제를 해결했습니다.

## 🤝 2. 다같이 확인할 사항 (피어 세션 토론 주제)

### 2.1. Well-Formed XML과 데이터 처리

-   **Well-Formed XML 이란?:** "형식이 잘 갖춰진" XML로, 단 하나의 최상위 엘리먼트, 모든 태그의 짝 맞춤, 올바른 중첩 등 엄격한 규칙을 따르는 문서를 의미합니다.
-   **XML을 사용하는 이유 (토론해보기):**
    -   **데이터의 구조적 표현:** 사람과 기계 모두가 이해하기 쉬운 형태로 계층적인 데이터를 명확하게 표현합니다.
    -   **플랫폼 독립성:** 순수 텍스트 기반이므로 거의 모든 시스템에서 데이터 교환 표준으로 사용될 수 있습니다.
    -   **확장성 및 검증:** DTD나 XSD 같은 스키마를 사용하여 데이터의 구조와 유효성을 사전에 정의하고 검증하여 데이터 무결성을 보장하기 용이합니다.

### 2.2. XML vs. JSON 비교 토론

| 구분 | XML (eXtensible Markup Language) | JSON (JavaScript Object Notation) |
| :--- | :--- | :--- |
| **장점** | - **스키마(DTD, XSD)를 통한 강력한 유효성 검증**<br>- **네임스페이스(Namespace)로 태그 이름 충돌 방지**<br>- **주석(Comment)을 공식적으로 지원** | - **문법이 단순하고 간결하여 가독성이 높음**<br>- **XML보다 훨씬 가벼워 네트워크 전송에 유리**<br>- **JavaScript에서 즉시 객체 변환 가능** |
| **단점** | - **태그 문법이 장황하여 파일 크기가 큼**<br>- **파싱이 JSON보다 상대적으로 복잡하고 느림** | - **주석을 공식적으로 지원하지 않음**<br>- **스키마를 통한 유효성 검증 기능이 상대적으로 미흡** |
| **결론** | 데이터의 **무결성, 유효성, 공식적인 문서 구조**가 중요할 때 (e.g., 정부 문서, 복잡한 설정 파일, SOAP 웹 서비스) | **빠른 속도, 가벼운 데이터 교환, 가독성**이 중요할 때 (e.g., RESTful API, 웹 애플리케이션) |
