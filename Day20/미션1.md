# **기능요구사항**

### **Top-down 설계**

다음과 같은 단계를 거쳐서 큰 것부터 작은 것까지 세부적으로 설계를 진행해야 한다. 각 단계별로 문서를 작성해서 gist에도 파일별로 업로드하거나 수정해야 한다.

### **1. 요구사항 분석**

아래 CPU 명령어와 CPU 요구사항을 읽고 분석해서 **`1.스펙.md`** 문서를 작성한다. **`1.스펙.md`** 파일에는 다음과 같은 내용을 포함해야 한다.

- 스펙: 요구사항을 담당 개발자 관점에서 필요한 것만 다시 정리해야 한다
- 상위 단위 타입과 데이터 구조: 이름이나 어떤 맥락에서 정의가 필요한지 설명한다. 모든 타입이 아니라 main 에서 바라볼 때 필요한 데이터 타입부터 정의한다.
- 전체적인 구조와 데이터 흐름을 도식화한 그림을 포함한다.

### **2. 모듈/타입 설계**

분석한 **`1.스펙.md`** 문서를 더 상세하게 설계하는 **`2.모듈설계.md`** 파일을 작성한다. **`2.OOOO모듈설계.md`** 문서에는 다음과 같은 내용을 포함해야 한다.

> 자바스크립트에서의 모듈과 코틀린/자바, 스위프트의 모듈 단위가 다르기 때문에 구현하는 단위-타입 정도를 부르는 일반적인 모듈의 의미로 해석하세요.
> 
- OOOO 모듈/타입 설계 : 모듈이나 타입별로 담당하는 역할과 책임을 작성한다.
    - 어떤 액션/동작을 담당하게 되는가
    - 각 액션/동작에서 필요한 데이터/값은 무엇인가
    - 액션/동작이 끝나면 바뀌는 데이터/값은 무엇인가
    - 액션/동작이 끝나고 전달해야 하는 데이터/값은 무엇인가
- 서브 타입과 데이터 구조: 내부에서 사용할 저 하위에 타입이나 데이터 구조
    - 값 범위: 저장하거나 전달하는 값의 범위를 정한다. 문자열인 경우는 최소/최대 길이도 정한다.
    - 값 존재여부: 없어도 되는지 있어도 되는지 중복 가능한지 고유한 지 등
    - 대표적인 테스트 케이스: 입력과 출력, 입력과 동작 확인에 대한 유형 설명

### **3. 소스 파일**

- 2단계에서 만든 모듈/타입 문서와 1:1로 소스 파일을 하나씩 만들어야 한다.
- 설계 단계에서 고민한 것을 바탕으로 구현한다.
- 만약 구현하면서 빠진 스펙이 생각났다면 작업 중이던 것까지 완성하고 다시 1-2단계로 되돌아서 추가해도 된다.
- 구현 과정에서는 테스트 코드를 적극 활용한다.

### **CPU 명령어**

다음과 같은 명령어만 실행하는 CPU를 설계하고 구현해야 한다.

**명령어 형식**

다음과 같은 명령어를 지원하는 CPU라고 가정한다.

| **INSTRUCTION** | **4bit** | **3bit** | **3bit** | **3bit** | **3bit** | **설명** |
| --- | --- | --- | --- | --- | --- | --- |
| **LOAD** | **0001** | **dst. Reg** | **base. Reg** | **000** | **offset. Reg** | **(base. Reg + offset. Reg) 주소 메모리 값을 읽어서 dst. Reg로 로드한다.** |
| **STORE** | **0011** | **src. Reg** | **base. Reg** | **000** | **offset. Reg** | **src. Reg 값을 (base. Reg + offset. Reg) 주소 메모리에 저장한다.** |
| **ADD** | **0111** | **dst. Reg** | **op. Reg** | **000** | **op. Reg** | **op. Reg 와 op. Reg 값을 덧셈(+) 연산해서 dst. Reg에 저장한다.** |
| **SUB** | **1010** | **dst. Reg** | **op. Reg** | **1** | **op. value(5bit)** | **op. Reg 와 op. Value 값을 뺄셈(-) 연산해서 dst. Reg에 저장한다.** |
| **MOV** | **1011** | **dst. Reg** | **op. Value(9bit)** |  |  | **op. Value 값을 dst. Reg에 저장한다.** |

> 각 Register는 3bit로 R1부터 R7까지 명시할 수 있다.
> 

R1 ==> **`001`**

R4 ==> **`100`**

R7 ==> **`111`**

**명령어 비트코드 예시**

**`LOAD R1, R2, R3`** ==> **`0001 001 010 000 011`**

**`STORE R5, R3, R4`** ==> **`0011 101 011 000 100`**

**`ADD R4, R2, R5`** ==> **`0111 100 010 000 101`**

**`SUB R4, R6, R7`** ==> **`1010 100 110 110 111`**

**`MOV R4, #250`** ==> **`1011 100 011111010`**

# **프로그래밍 요구사항**

### **CPU 요구사항**

- CPU 클래스를 다음과 같은 요구사항에 맞춰서 구현한다.
    - 기본 처리 단위 Word는 2바이트(16비트)를 기준으로 정한다.
    - Register는 PROGRAM-COUNT 역할을 하는 PC, 그 외에 일반적으로 사용할 수 있는 7개 R1-R7 레지스터를 포함한다.
    - ALU는 두 레지스터에 저장된 값 A, B를 계산해서 목적 레지스터에 저장한다.
    - ALU 계산에는 ADD 덧셈, SUB 뺄셈만 존재한다.
    - CPU 이외 메모리를 직접 구현할 필요는 없고, 기계어로 된 프로그램이 들어있다고 가정하고 구현한다.
        - 데이터 메모리 범위는 아래 예시에 맞춰서 특정한 주소 (0x00A2를 포함하는 범위)에 특정한 값이 들어있다고 가정한다. (간단하게 Map 또는 Dictionary로 주소와 값을 매핑해도 된다.)

### **필수 구현 메소드**

- **`fetch()`** 함수는 현재 PC 값에 해당하는 메모리에서 프로그램 명령어를 가져와서 리턴한다. PC 카운트를 +1 증가시킨다.
    - fetch에서 리턴한 명령을 execute에 넘겨준다.
- **`execute(Int16 IR)`** 함수는 전달한 명령어를 어떤 명령인지 분석해서 계산하거나 처리한다.
- 그 외 필요한 메소드는 스스로 판단한다.

# **예상결과 및 동작예시**

```
0x0000b MOV R4, 0x00A0
0x0010b MOV R5, 0x0002
0x0020b LOAD R1, R4, R5
0x0030b ADD R2, R1, R4
0x0040b SUB R3, R1, R2
0x0050b STORE R3, R4, R5

```

- 이 프로그램을 기계어(비트코드)로 표현바꾸고 CPU에 전달해서 실행한다
    - 프로그램을 기계어로 바꾸는 프로그램을 작성할 필요는 없다
    - 직접 비트코드를 넘겨줄 수 있으면 된다
